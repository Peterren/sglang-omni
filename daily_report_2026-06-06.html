⚡ 主题一 · 推理性能与解码路径

#692

[Higgs] Optimize Higgs async decode resolve output handling merged

作者 @Ratish1 · 合并于 2026-06-04 18:48 · 改动文件 1 个：higgs_tts/model_runner.py · 查看 PR
▾
📌 一句话：给 Higgs TTS 的「异步解码」收尾函数补上一个开关，让它在异步路径下不再把结果张量强行搬到 GPU，而是留在 CPU 上，省掉一次没必要的、还可能引发隐性同步的 GPU 操作。

🩹 它解决了什么问题

Higgs TTS 在「解码（decode）」阶段每一步都会产出一批 token（这里叫 codes / 声学码）。框架有两条收尾路径：

  • 同步路径（sync）：算出的 token 立刻要喂给下一步继续生成，所以必须放在 GPU 上。
  • 异步路径（async）：GPU 那边其实已经把「下一步真正需要的 codebook-0」单独发布出去了；这条收尾路径只负责把已经拷回 CPU 的结果整理成输出（给用户的音频），并不参与下一步计算。

问题在于：改动前，两条路径共用同一段收尾代码，而这段代码无条件地把整理好的 next_token_ids 创建在 GPU 上（用 ...next_token_logits.device 这个设备）。对异步路径来说，这等于「把本来在 CPU 的数据又搬回 GPU」——既浪费，又可能触发一次隐性的 CPU↔GPU 同步，拖慢吞吐。

🔧 具体做了什么改动

核心思路：给收尾函数 _decode_collect_host 增加一个仅限关键字的参数 next_token_device，由调用方决定结果该放哪。

# 改动前：设备写死，无论同步还是异步都放到 GPU
result.next_token_ids = torch.tensor(
    cb0_per_row,
    dtype=torch.long,
    device=result.logits_output.next_token_logits.device,  # ← 永远 GPU
)

# 改动后：由参数决定；None 就留在 CPU
if next_token_device is None:
    result.next_token_ids = torch.tensor(cb0_per_row, dtype=torch.long)  # CPU 张量
else:
    result.next_token_ids = torch.tensor(
        cb0_per_row, dtype=torch.long, device=next_token_device
    )

然后两个调用点按各自语义传值：

异步路径 post_decode_resolve
next_token_device=None
→ 结果留在 CPU，只做输出处理
同步路径 _collect_step_outputs_cg
next_token_device=result.logits_output.next_token_logits.device
→ 结果放 GPU，喂给下一步

函数签名里的 * 表示后面的 next_token_device 是「keyword-only 参数」——调用时必须写出参数名（next_token_device=...），不能靠位置传。这是一种防止调用方搞错参数顺序的工程小习惯。

📚 涉及的知识点（小白向）

Model Runner（模型运行器）
是什么：SGLang 里真正「把一批请求喂进模型、跑一次前向、拿到下一个 token」的执行单元。Scheduler 负责决定这一步跑哪些请求，Model Runner 负责真的去算。
为什么需要：把「调度策略」和「张量计算 + 设备管理」解耦，调度器不必关心 CUDA 细节。
本 PR 里：HiggsTtsModelRunner 的解码收尾逻辑就在这里，决定结果张量放 CPU 还是 GPU。
CUDA Graph（CUDA 图，代码里的 _cg / cg_buffers）
是什么：把一连串固定的 GPU 操作「录制」成一张图，之后一次性回放，而不是每步都由 CPU 逐条下发指令。
为什么需要：LLM 解码每步计算量小但步数极多，CPU 逐条下发 kernel 的「启动开销」会成为瓶颈；CUDA Graph 把这部分开销几乎抹平。类比：与其每次点菜都喊一句，不如把整套流程录成自动流水线一键播放。
本 PR 里：_collect_step_outputs_cg 就是 CUDA Graph 路径的收尾，它属于同步路径，所以结果要留在 GPU。
D2H 拷贝与「隐性同步」
是什么：D2H = Device(GPU)→Host(CPU) 的数据拷贝。GPU 计算是异步的——CPU 发完指令就继续往下走。但一旦你要把 GPU 结果搬到 CPU（或反过来建张量），常会强制等待 GPU 算完，这就是「同步点」。
为什么在意：同步点会让流水线「卡一下」，频繁同步会显著降低吞吐。
本 PR 里：注释里那句 one blocking D2H (sync path) 说的就是同步路径里那次会阻塞的拷贝；异步路径则刻意避免再把数据搬回 GPU，从而少一个同步隐患。
Codebook / Codebook-0（声学码本）
是什么：现代神经 TTS 不直接生成波形，而是先生成一串「离散声学 token」。这些 token 来自多个码本（multi-codebook，如 RVQ 残差量化），codebook-0 通常是最主干的那一路。
为什么需要：把连续音频压成离散 token，LLM 才能像生成文字一样「生成声音」，最后再由 vocoder 还原成波形。
本 PR 里：cb0_per_row 收集的就是每个请求这一步的 codebook-0 token；同步路径要拿它当「下一步的输入」，异步路径只拿它做输出整理。

💡 我能学到什么 / 延伸思考

  • 「让调用方决定副作用」是非常通用的设计：同一段核心逻辑，把「结果放哪个设备」这种与上下文相关的决定外提为参数，就能让一份代码同时服务两条语义不同的路径，避免复制粘贴。
  • 性能优化常常是「做减法」：这个 PR 没加任何新算法，只是删掉一次不必要的 GPU 张量创建。在推理框架里，识别并消除多余的设备拷贝/同步，是性价比极高的优化。
  • 前置知识建议：先搞懂 PyTorch 的 .device / .cpu() / CUDA 流（stream）与异步执行，再读「CUDA Graph 在 LLM 解码中的应用」，就能完全看懂这类 PR。
#691

[LLaDA2-Uni] Set default attention backend to flashinfer merged

作者 @btw616 · 合并于 2026-06-04 16:26 · 改动文件 1 个：llada2_uni/stages.py · 查看 PR
▾
📌 一句话：给 LLaDA2-Uni 这个模型的「thinker（思考/语言）阶段」把默认的注意力计算后端从框架默认值显式改成 flashinfer，只加了一行配置。

🩹 它解决了什么问题

原来构造该模型推理引擎时只设了 disable_cuda_graph=True，注意力后端用的是 SGLang 的默认值。但 LLaDA2-Uni 是一个扩散式语言模型（diffusion LLM, dLLM），它的注意力访问模式与普通自回归模型不同，默认后端不一定最优/最稳。显式钉死成 flashinfer，能拿到更好的内核性能与确定的行为，避免「换了环境默认值漂移」带来的不一致。

🔧 具体做了什么改动

改动前
overrides = {"disable_cuda_graph": True}
改动后
overrides: dict[str, Any] = {
    "attention_backend": "flashinfer",
    "disable_cuda_graph": True,
}

这是个 overrides 字典，会传给 build_sglang_server_args(...) 去覆盖默认的 server 参数。注意下面还有一行 overrides.update(server_args_overrides or {})——意味着这只是默认值，用户/上层仍可再覆盖。这是「合理默认 + 可覆盖」的经典配置模式。

📚 涉及的知识点（小白向）

Attention Backend（注意力后端）
是什么：Transformer 的核心是「注意力」运算。同一个数学公式可以有很多种 GPU 实现（FlashAttention、FlashInfer、Triton、Torch 原生…），这些可插拔的实现就叫「注意力后端」。
为什么需要：不同后端在不同 GPU、不同序列长度、不同批大小下的速度/显存差别巨大；做成可切换，框架就能因地制宜挑最快的那个。
本 PR 里：把 LLaDA2-Uni thinker 阶段固定为 flashinfer。
FlashInfer
是什么：一个专为 LLM 推理优化的高性能注意力/采样内核库，对各种 KV cache 布局、变长批、PagedAttention 等场景都有高度优化的 CUDA kernel。
为什么需要它：推理（尤其解码阶段）的注意力是 memory-bound（受显存带宽限制），FlashInfer 这类库通过精心设计的访存与分块把带宽吃满。
本 PR 里：作为该阶段的默认后端，换取更优、更确定的注意力性能。
多阶段架构里的「Thinker」阶段
是什么：SGLang-Omni 是「多阶段流水线（Multi-Stage Pipeline）」框架。一个 Omni 模型常被拆成若干阶段，比如「thinker（理解/生成语义）→ talker/vocoder（生成声音）」。stages.py 就是定义这些阶段如何拼装的地方。
为什么需要：把大模型按职责切分成阶段，可以分别用最合适的引擎、并行度、后端来跑，互不干扰（详见后面 #655、#647 也都围绕「阶段」展开）。
本 PR 里：函数名 create_sglang_dllm_thinker_executor_from_config 表明这是在为「dLLM thinker」这一阶段创建执行器，只调它的注意力后端默认值。
Diffusion LLM（扩散式语言模型，dLLM）
是什么：不同于「从左到右一个个吐 token」的自回归模型，扩散式 LLM 通过「从噪声/掩码逐步去噪/还原」的方式并行地确定多个 token。LLaDA2 系列就属于此类。
为什么需要：它有潜在的并行解码优势，但访存与计算模式和自回归不同，对底层内核的要求也不一样——这正是要专门指定后端、并 disable_cuda_graph 的原因。
本 PR 里：函数名里的 dllm 就是 diffusion LLM 的缩写。

💡 我能学到什么 / 延伸思考

  • 「显式优于隐式」：依赖上游默认值会让行为随版本/环境漂移。把关键配置显式钉死，是工程可复现性的基本功。
  • 一行代码的 PR 背后往往有一次实测对比。延伸可了解：不同 attention backend 的基准测试方法、FlashInfer vs FlashAttention 的差异、以及为什么扩散式模型要禁用 CUDA Graph（动态形状/动态步数难以录制成静态图）。

#686

Add support for pyAV-based mp3, aac, opus audio encoding merged

作者 @haojin2 · 合并于 2026-06-04 05:03 · 改动核心：client/audio.py + 新增单测 · 查看 PR
▾
📌 一句话：把 TTS 输出音频的编码主力从 pydub 换成 PyAV（FFmpeg 的 Python 绑定），支持 mp3/aac/opus，并在采样率不被编码器支持时自动重采样，pydub/WAV 作为兜底。

🩹 它解决了什么问题

模型生成的是浮点波形数组（numpy float32），但用户想要的是 .mp3/.aac/.opus 这类压缩格式。之前走 pydub，而 pydub 依赖外部 ffmpeg 可执行文件、对采样率限制处理不友好——比如 Opus 只接受 8k/12k/16k/24k/48k 这几种采样率，喂个 44100 进去就会报错。改动前没有「采样率不合法就自动调整」的机制，容易直接失败。

🔧 具体做了什么改动

① 定义每种格式的能力表（容器、可用编码器、合法采样率集合）：

PYAV_ENCODE_CONFIGS = {
    "opus": {"container": "ogg", "codecs": ("libopus","opus"),
             "valid_rates": {8000,12000,16000,24000,48000}},
    "aac": {"container": "adts", ...},
    "mp3": {"container": "mp3", "codecs": ("libmp3lame","mp3"), ...},
}

② 采样率不合法 → 自动重采样到最近的合法值（线性插值）：

if sample_rate not in valid_rates:
    nearest_rate = min(valid_rates, key=lambda r: abs(r - sample_rate))  # 找最接近的合法采样率
    audio = _resample_linear(audio, sample_rate, nearest_rate)         # 重采样
    sample_rate = nearest_rate

_resample_linear 的原理：用 np.interp 做线性插值。按原时长算出新长度，再在新的采样点上对旧波形插值——相当于「把一条曲线用不同密度的点重新采样」。

③ 用 PyAV 真正编码：开内存容器 → 添加音频流（依次尝试候选编码器）→ 把波形包成 fltp（float planar）格式的 AudioFrame → 逐帧 encode 并 mux（封装进容器）→ 最后用 encode(None) 冲刷（flush）编码器缓冲区。

frame = av.AudioFrame.from_ndarray(audio.reshape(1,-1), format="fltp", layout="mono")
frame.sample_rate = sample_rate
for packet in stream.encode(frame): container.mux(packet)
for packet in stream.encode(None): container.mux(packet)  # flush 残留

④ 优雅降级（graceful degradation）：整段包在 try/except 里，PyAV 失败就回退 pydub，再不行回退 WAV；FLAC 仍走 soundfile。所以无论环境缺哪个库，用户至少能拿到 WAV 而不是直接报错。

⑤ 新增 144 行单测，对每种格式都验证「魔数字节头」（如 Opus→OggS、MP3→ID3、AAC→0xFF 0xFx 同步字），并测试了重采样路径。

📚 涉及的知识点（小白向）

容器（container） vs 编码器（codec）
是什么：codec 决定「声音用什么算法压缩」（如 opus、aac、mp3）；container 决定「压缩后的数据装进什么文件盒子」（如 ogg、adts、mp3）。两者是正交的。
类比：codec 是「真空压缩衣服的方法」，container 是「装衣服的行李箱」。
本 PR 里：配置表里 codecs 和 container 分开写就是这个原因，比如 opus 用 ogg 容器。
采样率（sample rate）与重采样
是什么：每秒采多少个声音点，单位 Hz。24000 表示一秒 24000 个采样点。不同编码器只支持特定采样率。
为什么需要重采样：模型可能输出 44100，而 Opus 不收，就得换算到它支持的 48000/24000。
本 PR 里：_resample_linear 用线性插值做了一个轻量级（不依赖额外库）的重采样实现。
PCM / 浮点波形 / fltp
是什么：PCM 是「未压缩的原始采样值」。模型输出常是 float32（范围约 [-1,1]）。fltp=float planar，是 FFmpeg 编码器要求的内部样本格式。
为什么需要：编码器对输入样本格式有要求，喂错格式会失败；所以要先把 numpy 数组转成它认的 fltp/mono。
本 PR 里：AudioFrame.from_ndarray(..., format="fltp", layout="mono") 完成了这个适配。
优雅降级（Graceful Degradation）
是什么：当首选方案失败时，自动退到次优方案，保证「功能不挂、只是质量/格式降一级」。
为什么需要：推理服务部署环境千差万别（某些机器没装 ffmpeg/PyAV），健壮的服务不该因为缺个库就 500。
本 PR 里：PyAV → pydub → WAV 的三级兜底链。

💡 我能学到什么 / 延伸思考

  • 用「配置表 + 通用函数」消灭重复：三种格式的差异全部抽进 PYAV_ENCODE_CONFIGS，编码逻辑只有一份 _encode_with_pyav。新增一种格式只要加一行配置。
  • 测「魔数字节头」是验证二进制产物的好办法：不必解码整段音频，只看文件头几个字节就能确认格式对不对，测试又快又稳。
  • 延伸：了解 FFmpeg/PyAV 的 frame→packet→mux 流水线、为什么要 flush 编码器、以及高质量重采样（如 soxr、多相滤波）相比线性插值的差别。

#655

Add PCM audio streaming controls for Higgs TTS merged

作者 @Hayden727 · 合并于 2026-06-03 14:31 · 改动文件 15 个（+561 / −217）· 新增 models/tts_streaming.py · 查看 PR
▾
📌 一句话：新增一种「原始 PCM 字节流」的传输方式，并引入 initial_codec_chunk_frames 参数，让 TTS 第一个音频块可以「先小后大」——尽快吐出第一声、把首字延迟（TTFB）压到最低。

🩹 它解决了什么问题

流式 TTS 最关键的体验指标是首字节延迟（Time-To-First-Byte / 首声延迟）——用户点了「播放」，多久能听到第一个音。原来的流式有两个痛点：

  • 只支持 SSE（Server-Sent Events，文本事件流，里面塞 base64 音频），对实时音频播放器不够「裸」，多一层编解码开销。
  • vocoder 解码是按固定步幅（stride）成块产出的，第一块和后续块一样大。块越大，第一声来得越晚。

这个 PR 让「第一块」可以单独设成很小（甚至 1 帧），后续再恢复成稳态大块——首声秒出，整体吞吐基本不受影响。

🔧 具体做了什么改动

① 新增公共模块 models/tts_streaming.py：定义参数名常量与一个把请求参数「夹紧（clamp）」到合理范围的工具函数。

INITIAL_CODEC_CHUNK_FRAMES_PARAM = "initial_codec_chunk_frames"

def resolve_initial_codec_chunk_frames(params, *, steady_chunk_frames: int) -> int:
    """返回请求级的首块帧数，并夹紧到不超过稳态块大小。"""
    ...
    return min(frames, int(steady_chunk_frames))  # 首块不会比稳态块还大

② API 层（protocol.py / openai_api.py）：给 /v1/audio/speech 请求体加两个字段：

  • stream_format: "sse" | "audio"——选 "audio" 就走全新的「裸 PCM 字节流」。
  • initial_codec_chunk_frames: int | None——首块大小，带 ge=0（≥0）校验。

并加了一个模型级校验器，把「非法组合」在入口处直接挡掉：

@model_validator(mode="after")
def validate_stream_format(self):
    if self.stream_format == "audio":
        if not self.stream:
            raise ValueError('stream_format="audio" requires stream=true')
        if self.response_format.lower() != "pcm":
            raise ValueError('stream_format="audio" requires response_format="pcm"')
    return self

③ 服务端裸流处理 _speech_audio_response：它先消费流，直到拿到第一块有声音的 chunk，从中读出采样率，用它来设置响应头（X-Sample-Rate / X-Channels / X-Bit-Depth），然后把第一块和后续块作为纯字节连续吐出。这样客户端不用解析 JSON，直接拿 16-bit PCM 喂播放器。后续块若采样率突变会直接报错（保证流一致）。

④ 真正影响首声延迟的核心：vocoder_scheduler.py 的 _decode_delta。这里实现了「首块小、后续大」的逻辑：

改动前（每块同样大）
next_decode_rows = state.next_decode_rows or max(
    num_codebooks, self._stream_stride
)
改动后（首块可更小）
use_initial_chunk = (
  state.initial_codec_chunk_frames > 0
  and ... < steady_codec_frames
  and not state.has_emitted)   # 仅第一次
next_decode_rows = ... first_decode_rows
    if use_initial_chunk ... else 稳态

逻辑要点：只有「还没吐过任何块（not has_emitted）」且「首块设定值小于稳态值」时，才用小首块；吐完第一块后，_next_decode_rows_after_emit 立刻把步幅恢复到稳态 stream_stride。这是一种「预热式（warm-up）分块」策略。

📚 涉及的知识点（小白向）

Vocoder（声码器）与 Vocoder Scheduler
是什么：vocoder 把离散声学 token / 频谱还原成真实波形。SGLang-Omni 把它做成独立的一个阶段（stage），由 HiggsStreamingVocoderScheduler 调度。
为什么需要它：LLM 阶段产出的是 codes，不是声音；必须经 vocoder 才能听。把它单列一个阶段，就能独立控制它的「成块节奏」。
本 PR 里：首块大小的真正控制点就在 vocoder scheduler 的 _decode_delta。
流式输出 / SSE vs 裸字节流
是什么：SSE 是「服务器持续推文本事件」的 HTTP 机制，常用来逐块下发 base64 音频；裸字节流（media_type="audio/pcm"）则是直接把二进制音频顺着 HTTP body 流下去。
为什么需要裸流：少一层 base64 编解码与 JSON 包装，延迟更低、更适合直接喂音频播放管线；采样率等元信息改用 HTTP 响应头声明。
本 PR 里：stream_format="audio" 即裸 PCM 流，用 X-Sample-Rate 等头传元数据。
首字节延迟（TTFB）与「预热分块」
是什么：从发请求到收到第一个有效字节的时间。对语音/对话体验影响极大。
为什么「首块要小」：第一块越小，越早能产出 → 用户越早听到声音；之后再用大块保吞吐。这是延迟和吞吐之间的经典权衡。
本 PR 里：裸流默认 RAW_PCM_DEFAULT_INITIAL_CODEC_CHUNK_FRAMES = 1，即首块只要 1 帧就先发，把首声延迟压到极致。
参数夹紧（clamp）与入口校验
是什么：clamp 指把用户传入的值约束进合法范围（这里 min(frames, steady)）；入口校验（Pydantic model_validator）在请求一进来就拦截非法组合。
为什么需要：把「坏输入」挡在系统最外层，内部逻辑就能假设输入永远合法，少写一堆防御代码（fail-fast 原则）。
本 PR 里：首块不能超过稳态块（clamp），且 stream_format="audio" 必须搭 stream=true + pcm（校验器）。

💡 我能学到什么 / 延伸思考

  • 延迟优化的通用套路：把「第一份产出」做小做快，是流式系统（视频首帧、LLM 首 token、TTS 首声）共通的设计直觉。
  • 「契约 + 元数据下移到协议层」：用 HTTP 响应头承载采样率/位深，而不是塞进 body，是很干净的协议设计。
  • 把跨模型共享的常量/工具单列一个模块（tts_streaming.py）以便 Higgs、MOSS、Voxtral 等多个 TTS 模型复用——这正体现了「多阶段框架要面向多模型复用」的设计。
  • 延伸：读读 RVQ/delay-pattern 的解码（代码里有 reverse_delay_pattern）、以及流式 vocoder 的 holdback/lookahead 机制。

并发 / 扇出（concurrency / fan-out）与 Continuous Batching
是什么：并发=同时在飞的请求数。推理引擎会把多个并发请求动态拼成一个批一起算（continuous batching，连续批处理），以吃满 GPU。
为什么不是越高越好：批太大、工作量太小时，收益被排队/尾延迟吃掉；只有持续有大量请求时，高并发才把吞吐拉满。
本 PR 里：正是这个「小工作量下高并发反而慢」的现象，导致旧阈值失真。
WER（词错误率）与 RTF（实时率）
是什么：WER=语音识别结果与标准答案的错误比例（越低越好）；RTF=处理时长÷音频时长（<1 表示比实时还快）。
为什么这样分：WER 衡量「准不准」，RTF/吞吐衡量「快不快」。本 PR 强调：调并发只动「快不快」，几乎不动「准不准」。

是什么：ASR router 把进来的请求分发给多个后端 worker。「路由均衡度」衡量请求是否被均匀分配。
为什么重要：分配不均会让某些 worker 过载、整体吞吐下降，是排查性能问题的关键观测项。
