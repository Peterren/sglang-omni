#1047
Isolate Fish S2-Pro reference encoding from GPU stages
作者 Hayden727merged A · 多阶段流水线 合并 07-16 01:56 UTC
▸

github.com/sgl-project/sglang-omni/pull/1047

一句话: 把 Fish S2-Pro 的"参考音频编码"这段纯 CPU 计算从 GPU 进程里搬出来,单独放进一个进程跑,让 CPU 编码不再卡住 GPU 排队——冷启动场景 QPS 直接 +138.4%。

1 · 它解决了什么问题

Fish S2-Pro 是一个声音克隆 (voice cloning) 的 TTS 模型:你给它一段"参考音频"(想模仿的嗓音),它先把这段音频用一个 codec 编码成离散 token(叫 reference encoding / 参考编码),再拿这些 token 当条件去合成目标文本的语音。

痛点在于:这段"参考编码"是纯 CPU 计算,但它此前和 GPU 上的 TTS 引擎、声码器跑在同一个操作系统进程里。CPU 的 codec 会开一大堆原生线程去算,把 CPU 占满;于是负责给 GPU 喂活儿的调度线程抢不到 CPU,GPU 只能干等。PR 里的 stage trace 数据非常直观:

阶段Meanp95
preprocessing(参考编码)5.236 s7.460 s
tts_engine(GPU 生成)1.175 s—
vocoder(GPU 声码)0.199 s—

预处理占了 79.2% 的阶段驻留时间,而调度队列 p95 只有 31.6ms——说明瓶颈不是"排队",而是"CPU 编码本身把整台机器拖慢了,还顺带拖慢了 GPU"。

想象一个厨房里只有一个大灶台(GPU),旁边有个人在同一张操作台上剁一大盆肉馅(CPU 编码)。剁肉的动静太大、占满了台面,厨师连锅都端不上灶。这个 PR 做的事就是——把剁肉的人赶到隔壁房间(单独进程),操作台空出来,灶台立刻转起来。

2 · 具体做了什么改动

改动一:把 preprocessing 阶段放进独立进程

核心只有一个字的改动,在 config.py 里:

 StageConfig(
     name="preprocessing",
-    process="pipeline",
+    process="preprocessing",
     factory=f"{_PKG}.stages.create_preprocessing_executor",
     next="tts_engine",
 ),

在 SGLang Omni 里,每个 StageConfig 有一个 process 字段,决定这个阶段物理上跑在哪个操作系统进程。改之前,preprocessing 的 process="pipeline",意思是"和主流水线(tts_engine、vocoder)挤在同一个进程";改成 "preprocessing" 后,框架就会为它单独 fork 一个进程。tts_engine 和 vocoder 依然共享 GPU 0,不额外占卡。

改动二:给这个新进程的 CPU 线程池"上界"

光分家还不够——新进程里的 PyTorch 如果还是按整台机器(比如 224 核)去开线程,几个 worker 一叠加又会互相打架。所以 PR 加了一个函数专门给"intra-op 线程池"设上限:

def _configure_preprocessing_threads(worker_count: int) -> int:
    override = os.environ.get("OMP_NUM_THREADS", "").strip()
    if override.isdigit() and int(override) >= 1:
        requested = int(override)
        torch.set_num_threads(requested)       # ① 运维显式指定优先
        return requested

    cpu_count = (
        len(os.sched_getaffinity(0))           # ② 只看本进程能用的核
        if hasattr(os, "sched_getaffinity")
        else (os.cpu_count() or 1)
    )
    # 请求已经在 worker 线程间扇出;把共享 intra-op 池收窄,
    # 这样参考编码就不会饿死 GPU 流水线进程。
    intraop_threads = min(
        max(cpu_count // worker_count, 1),     # ③ 核数 ÷ worker 数
        _MAX_PREPROCESSING_INTRAOP_THREADS,    # ④ 再封顶到 8
    )
    torch.set_num_threads(intraop_threads)
    return intraop_threads

逐行拆解:

  • ① 如果运维在环境变量里显式写了 OMP_NUM_THREADS,一切以它为准——这叫"逃生舱口 (escape hatch)",保证优化不会剥夺人工控制权。
  • ② os.sched_getaffinity(0) 拿的是"本进程被允许使用的 CPU 核集合",而不是机器总核数。这很关键:因为进程被 numactl 绑核了,只看到自己那一份,数值才准。
  • ③④ 每个 worker 分到 总核数 / worker 数 个线程,至少 1,最多 8。"至少 1"防止除法归零,"最多 8"是实测出来的甜点值(PR 里做了 4/8/28 的线程扫描,8 吞吐最好)。

改动三:测试用真子进程验证线程上界真的生效

值得学习的一点:PR 不只测函数返回值,还起了一个真的 Python 子进程去读 torch.get_num_threads(),确认设置在真实进程里"落了地"、并且 OMP_NUM_THREADS 能压过 MKL_NUM_THREADS。因为线程池是全局副作用,只在主测试进程里 mock 是测不真的。

3 · 涉及的知识点(重点)

多阶段推理流水线 (multi-stage pipeline)
是什么: SGLang Omni 把一次推理拆成若干"阶段 (stage)",像流水线一样一站接一站:preprocessing → tts_engine → vocoder。每个阶段有自己的调度器 (scheduler) 和可配置的宿主进程/GPU 位置。
为什么需要: 多模态/语音模型的一次请求包含性质完全不同的工作——有的是 CPU 密集(音频预处理),有的是 GPU 密集(自回归生成)。拆成阶段后,可以分别给每个阶段挑最合适的硬件、并发度、缓存策略,还能让不同请求在不同阶段之间"重叠"起来跑。
本 PR 怎么用: 正是靠"阶段可以独立指定 process"这个能力,把 CPU 阶段挪到独立进程,实现 CPU 与 GPU 的真正并行重叠。
进程隔离 vs 线程 & Python GIL
是什么: Python 有个"全局解释器锁 (GIL)",同一进程内同一时刻只有一个线程能执行 Python 字节码。而底层原生库(codec、PyTorch 算子)会绕过 GIL 开自己的线程池抢 CPU。
为什么需要: 当 CPU 重活和 GPU 调度在同一进程,GIL 争用 + 原生线程抢核 会让"给 GPU 喂数据"的关键路径被拖住。放到独立进程后,两边有各自的解释器和 CPU 配额,互不干扰。
本 PR 怎么用: preprocessing 独立成进程,代价是多约 2.0 GiB 主机内存、线程从 700→808,换来 GPU 有效占用率(nonzero-SM duty)从 72.5%→88.9%。
intra-op 线程池 & OMP_NUM_THREADS
是什么: PyTorch 执行单个算子(如矩阵乘)时会用多线程并行,这叫 intra-op 并行,线程数由 torch.set_num_threads() / OMP_NUM_THREADS 控制。
为什么需要: 默认值往往是"机器全部核数"。当一台机器上跑很多进程/worker 时,每个都开满,总线程数爆炸(oversubscription),线程切换开销反而拖慢一切。
本 PR 怎么用: 按"本进程可见核数 ÷ worker 数、且 ≤8"来收窄,既保留并行度,又避免超额订阅。
NUMA & CPU 亲和性 (affinity)
是什么: 大服务器的内存分成多个 NUMA 节点,CPU 访问"本地节点"内存快、"远端节点"慢。sched_getaffinity 返回进程被绑定的核集合。
为什么需要: 把进程绑在离它 GPU/内存最近的 NUMA 节点上,减少跨节点访存延迟。
本 PR 怎么用: 线程数按"亲和性可见的核"来算,而不是全机核数,配合绑核部署才准确。

可复用的工程思想
  • "先量后改":PR 先用 stage trace 量出 preprocessing 占 79% 驻留,才动手——瓶颈定位靠数据不靠猜。
  • 隔离异构工作负载:凡是"性质不同的工作(CPU vs GPU、慢 vs 快)混在一个执行体里",就该考虑物理隔离。这是推理基础设施里非常通用的模式(prefill/decode 分离、encode/generate 分离都是同一思路)。
  • 优化要留逃生舱:自动设线程数,但 OMP_NUM_THREADS 永远压过自动值。
  • 验证要打到"真实副作用"层:全局线程池这种东西,用子进程做端到端验证,而不是只 mock。

想深入可以先补: 操作系统进程 vs 线程模型、Python GIL、OpenMP/MKL 线程模型、NUMA 架构、以及"生产者-消费者/流水线并行"的基本概念。

#1060
Bound Qwen3-Omni colocated stage CPU thread pools
作者 Hayden727merged A · 多阶段流水线 合并 07-16 15:30 UTC
▸

pull/1060

一句话: Qwen3-Omni 的 colocated 部署会启 8 个阶段进程,每个 PyTorch 进程都按整台 224 核机器开线程池,导致两个副本堆到 ~2940 个线程互相争抢。这个 PR 只做一件事:给这些进程默认设 OMP_NUM_THREADS=8 并关掉用不上的 tokenizer 并行——c32 吞吐 +49.3%,c64 +98.4%,线程数直接砍掉 56%。

1 · 它解决了什么问题

这是 #1047 的"同一个病、另一个模型"。Qwen3-Omni 是一个全模态 (omni-modal) 大模型(文本+语音+视频输入,能说话输出),它的 colocated(同机共置)部署会拉起 8 个独立的阶段进程。问题:

  • 每个 PyTorch 进程默认按"整台机器 224 核"去 sizing 自己的 OpenMP 线程池。两个单卡副本一叠加,主机线程飙到约 2940 个,在"启动侧 CPU 工作"上疯狂争抢,结果两张 GPU 的平均利用率只有 71–72%(卡在 CPU 上,喂不饱 GPU)。
  • preprocessing 阶段其实每次只处理一个 prompt,但 tokenizer 却初始化了一个"覆盖全机 CPU"的 Rayon 线程池——等于每个 CPU 加一个线程,却根本用不到批量分词。
好比一间只有 8 个工位的办公室,却发了 224 张门禁卡还每人配了 8 个助理。人挤人、助理挡道,真正干活的反而动弹不得。PR 做的是"按工位发卡"——每个进程只给 8 个线程。

2 · 具体做了什么改动

改动极小、极精准——只在 Qwen3-Omni 的 colocated 配置类上加了一组"环境变量默认值":

# 一个 colocated worker 启 8 个阶段进程。让每个 PyTorch 进程都
# 按整机 sizing OpenMP 池,会在多副本共享节点时超额订阅启动侧 CPU。
# preprocessing 每次只处理一个 prompt,全机 tokenizer Rayon 池只会加剧争用。
_COLOCATED_STAGE_ENV_DEFAULTS = {
    **_DEEPGEMM_PRECOMPILE_ENV_DEFAULTS,
    "OMP_NUM_THREADS": "8",           # 每进程 OpenMP 池封顶 8
    "TOKENIZERS_PARALLELISM": "false", # 关掉单-prompt 路径用不上的分词并行
}

class Qwen3OmniSpeechColocatedPipelineConfig(Qwen3OmniSpeechPipelineConfig):
    env_defaults: dict[str, str] = Field(
        default_factory=lambda: dict(_COLOCATED_STAGE_ENV_DEFAULTS)
    )

关键设计细节(在 PR 描述里明确强调):

  • "set-if-absent" 语义:这些叫 defaults——只在运维没显式设置时才生效,人工写的值永远赢。和 #1047 一个哲学。
  • 作用域收窄:只挂在 Colocated 配置类上;纯文本配置、disaggregated(分离式)语音配置完全不动。避免"一刀切"误伤别的部署形态。
  • 无新 CLI、不改 GPU 摆放/显存预算:纯 CPU 侧调优,零风险面。

测试里也顺手把契约锁死了——断言 colocated 配置的 env_defaults 恰好等于这三个键,防止以后有人误删。

3 · 涉及的知识点

colocated(共置)vs disaggregated(分离)部署
是什么: colocated = 把一个模型的多个阶段/组件挤在同一台机器(甚至同一张卡)上;disaggregated = 把不同阶段拆到不同机器/卡上独立扩缩容。
为什么需要: colocated 省机器、通信开销小,适合中小规模;disaggregated 灵活、可对瓶颈阶段单独加卡,适合大规模。二者的 CPU 争用特征完全不同,所以本 PR 的默认值只对 colocated 生效。
本 PR 怎么用: colocated 因为 8 进程挤一台机才有线程超额订阅问题,故仅收窄它。
线程超额订阅 (thread oversubscription)
是什么: 活跃线程总数远超物理核数。操作系统被迫频繁做上下文切换,CPU 时间花在"切换"而非"计算"上。
为什么需要关注: 多进程 × 每进程满线程,是超额订阅最常见的成因。表现就是"CPU 看着很忙但吞吐上不去、GPU 喂不饱"。
本 PR 怎么用: 2940→1284 线程(-56.3%),两个 preprocessing 进程不再各自膨胀到约 300 线程。
Tokenizer 的 Rayon 并行 (TOKENIZERS_PARALLELISM)
是什么: HuggingFace tokenizers 库用 Rust 的 Rayon 做多线程分词,默认对大批文本并行。
为什么需要关掉: 只有"一次分很多条文本"时并行才划算。这里 preprocessing 每次只有一个 prompt,并行池纯属白开线程、只添争用。
本 PR 怎么用: 设 false,单独再省掉约 740 个线程,吞吐不降反升。

4 · 我能学到什么 / 延伸思考

可复用的工程思想
  • "默认值也是接口":一个好的框架应该给不同部署形态各自合理的默认值,而不是让每个用户去踩坑后手动调环境变量。把知识固化进配置类,是把"运维经验"产品化。
  • 改动面 = 风险面:98% 的吞吐提升,代码只加了 4 行 + 作用域限定。真正高杠杆的优化常常是"发现一个错误的默认值"而非"写一个复杂新算法"。
  • 用测试锁契约:把"应该等于哪三个键"写进单测,防止未来回归。

想深入可以先补: OpenMP / Intel MKL 线程模型、上下文切换成本、HuggingFace tokenizers 的并行机制,以及 #1047 里的进程隔离——两者是"CPU 不要挡 GPU"这条主线的一体两面。

#1055
Reuse idle vocoder lanes for MOSS-TTS Local streaming
作者 Hayden727merged B · 声码器 & CUDA Graph 合并 07-17 05:20 UTC
▸

pull/1055

一句话: MOSS-TTS Local 的声码器有个固定宽度 16 的 CUDA Graph,却被切成"8 个流式槽 + 8 个离线槽";纯流式服务时那 8 个离线槽白白空着。这个 PR 把它重划成"15 流式 + 1 离线",并允许离线活儿临时借用空闲流式槽——不增显存、不增卡,c32 吞吐 +22.1%、首帧延迟(TTFC) -41.8%。

1 · 它解决了什么问题

先建立几个概念的画面感。声码器 (vocoder) 是 TTS 的最后一站:模型先生成离散的音频 codec token,声码器再把这些 token 解码成真正能听的 PCM 波形。为了快,MOSS 把声码器的解码过程用 CUDA Graph 预先"录制"成一个固定 batch 宽度=16 的持久会话(persistent session)。

但这 16 个"车道 (lane/slot)"被静态切成两半:8 个给流式 (streaming) 请求、8 个给离线 (offline) 一次性解码。问题来了——在正常的"只做流式"的服务里,那 8 个离线车道永远没人用,但它们占着 CUDA Graph 的宽度、显存都分好了,live 请求却碰不到。

更糟的连锁反应:当并发超过 8,抢不到流式槽的请求只能攒着所有 codec 行、到最后一次性 flush 解码整段话。这种大的"最终 flush"经常超出 CUDA Graph 捕获的帧范围(T=1..25),掉回慢速的 eager 执行。于是吞吐卡在一个平台期,而一半的图车道还闲着。

一条 16 车道的高速公路,硬性规定左边 8 道只走小轿车(流式)、右边 8 道只走货车(离线)。今天全是小轿车,右边 8 道空荡荡,左边却堵得水泄不通,还逼着堵住的车绕去走坑洼的辅路(eager)。这个 PR 说:货车平时只留 1 道应急,其余车道谁空谁上,轿车能全用起来。

2 · 具体做了什么改动

① 重划车道:15 流式 + 1 离线(总宽仍是 16)

-    stream_slots: int = 8,
+    stream_slots: int = 15,
...
# 纯非流式流量会关掉空闲会话、走打包批处理路径。
# 只留一条溢出车道保证流式在跑时离线也有进展,
# 而不是为这种情况把每张流式 CUDA Graph 的一半都预留掉。
-    self._offline_slots = max(int(max_batch_size), 1)
+    self._offline_slots = _SESSION_RESERVED_OFFLINE_SLOTS  # = 1

注意:这不增加 CUDA Graph 宽度、不加 GPU、不加显存——只是把同样的 16 个车道重新分配,把浪费的 7 个从"离线专用"变成"流式可用"。

② 让离线活儿"借"空闲流式槽,并且借得安全

这是本 PR 最精彩的部分:decode_offline 现在会计算"我需要几个槽、能借几个空闲流式槽",借来用完必须归还,归还前要 reset 状态:

reserved = list(range(self._stream_slots, self._batch_size))  # 那 1 个离线专用槽
requested_wave_size = min(max(int(max_batch_size), 1), len(codes_list))
borrow_count = min(
    max(requested_wave_size - len(reserved), 0),
    len(self._free_stream_slots),          # 只借当前空闲的流式槽
)
borrowed = self._free_stream_slots[-borrow_count:] if borrow_count else []
if borrowed:
    del self._free_stream_slots[-borrow_count:]  # 借走  移出空闲池
available = reserved + borrowed
...
decode_succeeded = False
try:
    # ... 用 available 这些槽逐"波"解码 ...
    decode_succeeded = True
finally:
    if borrowed:
        try:
            self._reset_slots(borrowed)        # ③ 归还前必须复位状态
        except Exception:
            logger.exception("...quarantining them")  # ④ 复位失败  隔离该槽
            if decode_succeeded:
                raise
        else:
            self._free_stream_slots.extend(borrowed)  # ⑤ 复位成功才放回池

逐块理解这段"fail-closed(失败即关闭)"的安全设计:

  • ①借用记账:借走的槽立刻从 _free_stream_slots 删掉,这样它在整个解码生命周期里不会被别人再抢(独占)。
  • ③归还必复位:声码器每个槽内部维护着"流式状态 (streaming state)"——上一步解码的隐状态。借用后必须 reset 清干净,否则下一个流式请求会读到脏状态、音频出错。
  • ④复位失败=隔离 (quarantine):如果 reset 抛异常,这个槽不放回池子——宁可永久损失一个车道,也不把一个状态可疑的槽再给别人用。这就是"fail-closed":出错时倒向安全一侧。
  • ⑤只有复位成功才归还:干净的槽才回到 _free_stream_slots。
  • 另外 acquire/release 加了 _stream_slots_in_use 集合显式跟踪租约,重复释放/非法释放会直接 raise——不让"悄悄的双重释放"埋雷。

③ 每个流式泵每请求只取一行,守住首帧边界

还加了一个开关 _stream_chunk_batch_distinct_requests:在合并(coalesce)排队的 codec 行时,每个请求每轮只收一行,遇到同一请求的第二行就停下、把它推回队首。这样即使某个请求积压了一堆行,也不会一口气把它的初始 chunk 撑大,从而保住配置的"首帧尽早发出 (low TTFC)"的边界。默认只对 MOSS 开,别的调度器不受影响。

3 · 涉及的知识点(重点)

CUDA Graph
是什么: 把一串 GPU 操作(kernel 启动)预先"录制"成一张图,之后一次性"回放"。省掉了每次逐个启动 kernel 的 CPU 开销。
为什么需要: 小算子很多时,CPU 逐个发射 kernel 的开销会成为瓶颈(launch overhead)。CUDA Graph 把这块开销几乎归零,对 TTS 这种"每帧一小步"的解码特别有效。
为什么有"固定宽度"限制: 图是按固定形状(batch=16、帧数 T=1..25)录制的。超出这个范围就没法回放,只能掉回一条一条发射的 eager 模式,慢很多。本 PR 的痛点正是"大 flush 超出捕获帧范围掉 eager"。
本 PR 怎么用: 不改图的宽度(还是 16),只是把车道用满,并通过"每请求一行 + 步长封顶到 chunk 帧"让更多解码落在图捕获范围内——日志显示图化步骤从 99.3% 升到 99.8%。
流式 (streaming) vs 离线 (offline) 解码
是什么: 流式=边生成边一小块一小块地吐音频(用户很快听到第一声);离线=整段生成完再一次性解码返回。
为什么需要: 流式的核心指标是 TTFC(Time To First Chunk,首帧时间)——用户等多久能听到声音。离线更看整体吞吐。
本 PR 怎么用: 让离线借流式的空闲槽,同时用"每请求一行"守住流式的 TTFC——两个目标兼得,c32 的 TTFC -41.8%。
持久会话 & 槽 (slot / lane) 状态
是什么: 声码器把一个 batch=16 的 CUDA Graph 会话常驻显存,每个 batch 位置是一个"槽";每个槽内部有跨步(step)累积的流式隐状态。
为什么需要: 常驻会话避免每次重新捕获图的开销;但槽是有状态的,复用前必须 reset,否则串音/出错。
本 PR 怎么用: 借用/归还槽时严格 reset + 隔离,保证有状态资源被安全复用。
位级等价验证 (bit-identical)
是什么: 用 SHA-256 校验优化前后单请求的流式输出逐字节完全一致(sha256=a65fa4f3...)。
为什么需要: 这是一个"纯性能重构"——没动任何模型权重、采样数学、codec 数学。位级一致是"我没改变行为、只改变了速度"的最硬证据。
本 PR 怎么用: 配合 218 个单测 + 失败注入测试,证明借槽/复位/隔离逻辑既快又不改音质。

4 · 我能学到什么 / 延伸思考

可复用的工程思想
  • 静态分区是资源浪费的常见根源:"8+8 固定切分"看似公平,实则在偏斜负载下浪费一半。改成"保留最小份额 + 空闲可借"(reservation + work-stealing)是通用解法,和线程池/连接池的弹性借用同构。
  • 有状态资源的借用要 fail-closed:借得出去,更要还得干净;还不干净就隔离,绝不把可疑状态复用。这套 acquire/lease/reset/quarantine 是写并发资源池的范式。
  • 性能重构要有"零行为改变"的铁证:bit-identical 哈希 + 失败注入单测,比"跑了几个 case 没崩"可信得多。
  • 理解硬件的"快路径边界":CUDA Graph 有捕获范围,优化的一大半是"想办法让更多工作落在快路径内",而不是硬扩图。

想深入可以先补: CUDA Graph 原理与 kernel launch overhead、TTS 里 codec/vocoder 的角色、TTFC/RTF 等流式指标、并发资源池(acquire/release/lease)的设计模式。

Higgs: route ad-hoc reference codes through ReferenceEncodeService
作者 luojiaxuanmerged C · 共享服务 & 缓存 合并 07-16 15:54 UTC

pull/1051

一句话: 把 Higgs 最后一处"手搓的参考编码缓存"替换成全仓库共享的 ReferenceEncodeService。至此 fish / qwen3-tts / moss-local / higgs 四个模型全部改用同一套缓存服务——统一了缓存键、单飞去重、字节级 LRU 和统计日志。

1 · 它解决了什么问题

每个 TTS 模型都要缓存"参考音频→codec token"的结果(同一段参考音频没必要反复编码)。以前各模型各写各的缓存逻辑(Higgs 用 StageOutputCache),导致:重复代码、缓存键不统一、没有并发去重(两个请求同时来同一段音频会编码两次)、也没有统一的命中率统计。这个 PR 是 M4a 迁移清单的收尾——把 Higgs 也接到共享服务上。

四个部门各自建了一个小仓库存同样的货,盘点口径还不一样。现在改成一个中央仓库,统一编号(内容哈希)、统一防止"两个人同时下单同一件货各进一批"(单飞),统一按体积淘汰旧货(字节级 LRU)。

2 · 具体做了什么改动

关键在于"内容哈希做键"。参考编码的缓存键用的是预处理阶段就算好的波形内容哈希 state.reference_code_cache_key。这意味着——只要参考音频的内容相同,不管它来自文件路径、字节流还是 base64、也不管请求 ID 是什么,都会命中同一个缓存条目。内容键不会过期(内容变了哈希就变了)。

做法是给共享服务提供一个 Higgs 专用的 hook(钩子),把原来内联的编码逻辑原封不动搬进去:

class _HiggsReferenceEncodeHook(ReferenceEncodeHook[...]):
    def cache_key(self, item) -> ReferenceEncodeKey | None:
        if item.content_key is None:
            return None            # 没内容键 → 不缓存,直接透传(保持旧行为)
        return ReferenceEncodeKey(
            model_id=self._model_identity, model_revision="",
            encoder_id="higgs_codec_delayed",
            encoder_config_hash=f"nq{self._num_codebooks}",
            artifact_kind="reference_codes",
            input_key=item.content_key,   # ← 波形内容哈希
        )

    def encode_one(self, item) -> torch.Tensor:
        ref_codes_TN = self._codec.encode_reference(item.waveform, sample_rate=24000).to(torch.long)
        if ref_codes_TN.ndim != 2 or ref_codes_TN.shape[1] != self._num_codebooks:
            raise ValueError(...)     # 形状不对就大声报错(fail-loud)
        return apply_delay_pattern(ref_codes_TN)  # 编码数学一字未改

    def store_artifact(self, a) -> torch.Tensor:
        return a.detach().to("cpu", torch.int32)  # 存:int32 省内存
    def load_artifact(self, s) -> torch.Tensor:
        return s.detach().clone().to(torch.long)  # 取:转回 long 给下游

这个 hook 把一件事拆成四个明确职责:算键 (cache_key)、算值 (encode_one)、怎么存 (store)、怎么取 (load)。共享服务负责"命中就返回、没命中就调 encode_one、并发同键只算一次、按字节淘汰"这些通用机制,模型只需填四个方法。

另一处巧思是"两级缓存分工":上传的具名音色 (uploaded voice) 仍走"带版本号的 speaker 缓存"(它的失效契约是"重新上传"),而所有临时 (ad-hoc) 参考走内容键的共享服务;speaker 缓存未命中时,编码也会顺便把内容键条目"焐热",实现"跨名字复用"。存 int32、取回 long 的往返也被单测严格校验。

3 · 涉及的知识点

内容寻址缓存 (content-addressed cache)
是什么: 用"数据内容的哈希"当键,而不是用文件名/路径/请求 ID。
为什么需要: 内容相同则键相同,天然去重、天然不会过期(内容变→哈希变→自动是新键)。Git 的对象存储、Docker 层缓存都是这个思路。
本 PR 怎么用: 波形内容哈希做键,让同一段参考音频无论以什么形式、什么请求进来都只编码一次。
单飞 (single-flight) 去重
是什么: 多个并发请求要同一个键的结果时,只让"第一个"真正去计算,其余请求等它算完共享结果。
为什么需要: 冷启动瞬间可能有 N 个请求同时请求同一段新参考音频。没有单飞就会编码 N 次,既浪费又可能击穿(cache stampede)。
本 PR 怎么用: 共享服务内置单飞,同内容并发只 encode 一次。
字节级 LRU 淘汰
是什么: LRU=最近最少使用淘汰。"字节级"指按缓存占用的实际字节数(而非条目个数)来设上限。
为什么需要: 每段参考音频编码后的张量大小差异很大,按"条数"限容易内存失控;按字节封顶才可控。存成 int32 也是为了缩小占用。
本 PR 怎么用: max_bytes 设上界,张量以 CPU int32 存储。
Hook / 策略模式 (Strategy Pattern)
是什么: 把"通用流程"和"可变细节"分开:服务定义流程骨架,模型通过实现 hook 接口注入自己的细节。
为什么需要: 四个模型的编码数学不同,但"查缓存/去重/淘汰/统计"完全一样。用 hook 抽出共性,消除四份重复实现。
本 PR 怎么用: ReferenceEncodeHook 定义 cache_key/encode_one/store/load 四点,Higgs 只填这四点。

4 · 我能学到什么 / 延伸思考

可复用的工程思想
  • "收口 (consolidation)"迁移法:四个模型逐个(fish→qwen3-tts→moss→higgs)迁到同一服务,每个 PR 只搬一个、保持行为等价、加针对性测试。大重构拆成可验证小切片,是控制风险的正道。
  • 内容寻址是去重的银弹:只要能对输入取稳定哈希,缓存键、去重、失效三件事一次解决。
  • 用 hook 分离"机制"与"策略":通用机制(缓存/单飞/LRU)写一次,模型差异用接口注入。
  • fail-loud vs fail-closed:这里 codec 形状不对立刻报错(fail-loud),因为那是编程/权重错误,越早暴露越好——和 #1055 借槽时的 fail-closed 是"按错误性质选失败姿态"。

新增一个 _InferencePromptEncoder,只累积推理需要的三样张量,把"消息对象 → 编码 → 取出需要的字段"这条绕路,变成"直接一段段追加":

class _InferencePromptEncoder:
    def append_text(self, text: str) -> None:
        tokens = torch.tensor(self._tokenizer.encode(text), dtype=torch.int)
        self._token_segments.append(tokens)
        self._vq_mask_segments.append(torch.zeros_like(tokens, dtype=torch.bool))  # 文本段 mask=0

    def append_vq(self, codes: torch.Tensor) -> None:
        codes = codes.clone().to(torch.int)
        tokens = torch.tensor(self._tokenizer.convert_tokens_to_ids(
            [SEMANTIC_TOKEN_TEMPLATE.format(i=code) for code in codes[0].int()]  # 第一个码本 → 语义 token
        ), dtype=torch.int)
        self._token_segments.append(tokens)
        self._vq_mask_segments.append(torch.ones_like(tokens, dtype=torch.bool))   # VQ 段 mask=1
        self._vq_parts.append(codes)          # 完整码本行留存

    def finish(self) -> dict:
        return {
            "input_ids": torch.cat(self._token_segments, dim=0),
            "vq_mask_tokens": torch.cat(self._vq_mask_segments, dim=0),
            "vq_parts": self._vq_parts,
        }

然后 build_prompt 直接、显式地拼出聊天段落,不再构造 Message/Conversation 对象:

encoder.append_text(f"{IM_START_TOKEN}system\n")
encoder.append_text("convert the provided text to speech reference...")
for ref in references:
    if ref.text: encoder.append_text(f"<|speaker:{speaker}|>{ref.text}")
    ...
if all_codes:
    encoder.append_vq(torch.cat(all_codes, dim=1))  # 多个参考沿时间维拼接
encoder.append_text(f"{IM_END_TOKEN}\n")
# ... user 段 / assistant <|voice|> 段同理 ...
return encoder.finish()

怎么保证"只删臃肿、不改行为"?PR 加了参数化单测,覆盖无参考 / 单参考 / 多参考 / 纯文本 / 纯码五种 prompt,对每一段的 segment、token、mask、dtype、VQ 布局逐一断言;还测了零长度 VQ 张量和非法 rank/码本数报错。Accuracy 测试显示 corpus WER 甚至从 1.10% 微降到 1.00%(即输出未劣化)。

3 · 涉及的知识点

VQ (Vector Quantization,向量量化) & codec token
是什么: 把连续的音频波形压成一串离散整数("码"),每个码指向码本 (codebook) 里的一个向量。语音大模型正是把音频当成"离散 token 序列"来处理,才能和文本 token 统一用 Transformer 建模。
为什么需要: Transformer 天生处理离散 token。VQ 把音频"token 化",让语音能像文本一样被自回归生成。多码本 (num_codebooks) 是为了在有限码表下表达更丰富的音频细节。
本 PR 怎么用: 参考音频的 VQ 码进 prompt;第一个码本映射成"语义 token"(<|semantic:i|>),完整码本行存在 vq_parts 里给模型。
多模态 prompt 中的 mask
是什么: vq_mask_tokens 是一个和 token 序列等长的布尔数组,标记"哪些位置是 VQ 音频 token、哪些是普通文本 token"。
为什么需要: 文本 token 走词嵌入表,音频/VQ token 要走另一条 embedding 路径。模型靠这个 mask 知道每个位置该用哪套嵌入——这是"多模态输入对齐"的关键机制。
本 PR 怎么用: 文本段追加时 mask 填 0,VQ 段追加时 mask 填 1,精确保持原有语义边界。
聊天模板 (chat template) & 特殊标记
是什么: <|im_start|>/<|im_end|>/<|voice|> 等特殊 token 用来切分 system/user/assistant 角色和模态。
为什么需要: 模型是在这种固定格式上训练的,推理时必须逐字节对齐同样的模板,否则模型"看不懂"。
本 PR 怎么用: 用显式字符串拼出与原来完全相同的段落,并用单测逐段断言,保证模板不走样。
推理路径 vs 训练路径的分离
是什么: 训练需要 loss labels、可视化、数据增强等一大堆东西;推理(serving)只需要"构造输入→前向"。
为什么需要: 把训练专用代码留在推理路径上,会带来体积、依赖、维护和潜在 bug 面。生产服务应只保留自己真正需要的最小面。
本 PR 怎么用: 砍掉训练用的 Conversation 框架,只留推理专用编码器,减少 721 行生产代码面。

  • 删代码是最好的重构,但要有等价证据:净删 721 行的底气来自"逐 token 断言 + WER 未劣化"。删得越狠,越要证明行为不变。
  • 让数据流"直白可读":从"构造对象→encode→再挑字段"改成"一段段 append",代码意图一眼可见,也更好测。
  • 和 #1057、#1069 是同一主题(D):把推理路径上的臃肿/vendored/死代码持续剥离,让生产面收敛到最小。

1 · 它解决了什么问题

一张现代 GPU(H100/H200)显存很大,单个 TTS 模型往往吃不满。于是可以在同一张卡上并排跑多个模型副本 (data parallelism, DP) 来提高吞吐,再用 CUDA MPS 让这些副本真正并发共享 GPU。但这套部署很难调对,尤其是KV cache 的容量分配:

关键陷阱在 PR 文档里讲得很透彻:相同的 --mem-fraction-static 标志并不意味着相同的 KV 容量。因为副本是顺序启动的,先启动的副本已经占了一部分显存,后启动的看到的"空闲显存"更小,于是即使标志一模一样,后者算出来的 KV token 数也更少(文档里的真实例子:三个 mf=0.27 副本分别拿到 97503 / 53149 / 20961 个 KV token,差了近 5 倍!)。

三个人先后进同一个自助餐厅按"拿盘子容量 27%"取餐。第一个人来时菜是满的,拿得多;第三个人来时菜快没了,同样的"27%"却拿得少得多。所以不能只说"大家都拿 27%",必须规定"每人最终盘子里必须正好这么多",不够就报错。

2 · 具体做了什么改动

① 把"档位"变成可复用的 YAML 配置

新增两个 profile 文件,把"可移植的模型/运行时设置"和"主机专属的 GPU/CPU 摆放"分开。例如 H200 DP8:

# examples/mps_dp/configs/higgs_h200_dp8.yaml
config_cls: HiggsTtsPipelineConfig
model_path: bosonai/higgs-tts-3-4b
stage_overrides:
  tts_engine:
    runtime:
      sglang_server_args:
        mem_fraction_static: 0.85
runtime_overrides:
  tts_engine:
    server_args_overrides:
      max_total_tokens: 30000       # 每副本 KV 上限(H100 档是 100000)
      max_running_requests: 64      # 准入上限(不是并发保证)
      cuda_graph_max_bs: 64

PR 特别点明:H200 的 30000 KV 池其实小于"64 个请求每个生成 2048 新 token"的最坏需求。所以 max_running_requests=64 是一个准入天花板 (admission ceiling),不是"64 个长请求能同时解码"的保证。池满了 SGLang 会撤回 (retract) 请求、放回等待队列,等 KV 腾出来再继续。这是一个很重要的"承诺 vs 尽力"的区分。

② 启动脚本从配置解析 KV 上限,并逐副本精确校验

新增的 config.py 通过 ConfigManager 找到"generation 阶段"配置的 max_total_tokens;而 launch.sh 在每个副本健康后,从日志里抓出它实际解析到的 KV token 数,不等于期望值就直接 die:

resolved_tokens=$(grep -m1 -oE '#tokens:[[:space:]]*[0-9]+' "$log" | grep -oE '[0-9]+$')
if [ "$n" -gt 1 ]; then
  [ -n "$resolved_tokens" ] || die "replica $i healthy but resolved KV capacity missing"
  [ "$resolved_tokens" = "$expected_max_total_tokens" ] \
    || die "replica $i resolved $resolved_tokens KV tokens; expected $expected..."
fi

换句话说:因为"内存 profiling 不会跨进程协调 KV 分配",脚本强制要求 N>1 时每个副本都解析出完全相同的 KV 容量,否则拒绝启动。这把上面那个"顺序启动导致容量不均"的隐患变成了一个显式的、启动即失败的硬检查。同时保留了老的 MODEL + MAX_TOTAL_TOKENS 环境变量接口,向后兼容。

3 · 涉及的知识点(重点)

KV Cache(键值缓存)
是什么: Transformer 自回归生成时,每生成一个 token 都要用到前面所有 token 的 Key/Value 向量。把它们缓存下来避免重复计算,就是 KV cache。它是推理时最主要的显存消耗者之一。
为什么需要: 没有它,每生成一个新 token 都要把整段历史重算一遍,慢到不可用。有了它,生成第 N 个 token 只需算第 N 个。代价是显存——序列越长、并发越多,KV 占用越大。
本 PR 怎么用: max_total_tokens 就是 KV 池能容纳的总 token 数上限,直接决定能并发多少/多长的请求。
CUDA MPS(Multi-Process Service,多进程服务)
是什么: NVIDIA 的一个机制,让多个进程的 CUDA kernel 能真正并发共享一张 GPU(而不是默认的时间片轮转)。
为什么需要: 单个 TTS 模型常吃不满 GPU 的算力和显存。多副本 + MPS 能把利用率压上去,提高整卡吞吐。
本 PR 怎么用: 脚本启一个私有 MPS daemon,顺序拉起 N 个副本、逐个等健康、验证 MPS 挂载,H200 上 8 个副本全部挂上。
数据并行 (Data Parallelism, DP) —— 这里是"同卡 DP"
是什么: 跑多个完整的模型副本,把请求分散到各副本上。和"张量并行 (TP)"不同:TP 是把一个模型切开放到多卡协作算一次前向;DP 是每个副本都是完整模型、各自独立处理不同请求。
为什么需要: 当模型能装进单卡、瓶颈是吞吐时,DP 比 TP 更简单高效(没有跨卡通信)。这里更进一步——多个副本挤在同一张卡上,靠 MPS 并发。
本 PR 怎么用: H100 同卡 3 副本、H200 同卡 8 副本,每副本独立 KV 池,请求不在副本间分发(每个副本自己收自己的)。
mem-fraction-static & 请求撤回 (retraction)
是什么: mem_fraction_static 是"启动时按当前空闲显存的百分比预留给模型权重+KV 池"。retraction 是池满时把正在跑的请求踢回等待队列的机制。
为什么需要: 前者决定 KV 池大小,但因"顺序启动看到的空闲不同"而不可靠;后者保证池不够时系统不崩、只是降速(准入 ≠ 并发保证)。
本 PR 怎么用: 两档都用 0.85;并用"精确 KV 校验"绕开 mem-fraction 的不确定性,用 retraction 兜住 30000 池小于最坏需求的情况。

4 · 我能学到什么 / 延伸思考

可复用的工程思想
  • 把"隐性假设"变成"显式断言":大家默认"相同 flag = 相同容量",而现实不是。好的工程把这种隐性假设做成启动即失败的硬检查——错误在部署第一秒暴露,而不是压测时才发现容量不均。
  • 配置分层:可移植的 vs 主机专属的:模型/运行时设置进 YAML(可复用),GPU/CPU 摆放进环境变量(主机相关)。这种分层让"经过验证的配方"能被别人安全复用。
  • 区分"准入 (admission)"与"保证 (guarantee)":max_running_requests=64 只是准入上限,靠 retraction 兜底。理解这层区别,才不会误以为"设了 64 就能同时并发 64 个长请求"。
  • 档位要标注"验证边界":文档反复强调"这是特定配置下的验证值,不是通用默认;换模型/卡/副本数都要重算"。给出数字的同时给出适用范围,是负责任的性能工程。

想深入可以先补: KV cache 与 PagedAttention、CUDA MPS、数据并行 vs 张量并行、SGLang 的显存 profiling 与 max_total_tokens 解析、以及"准入控制 / 背压 (backpressure)"的系统设计概念。

#1057 · Use upstream Transformers Higgs tokenizer   D · SandyLuXY · pull/1057

做了什么: SGLang-Omni 之前因为老版 Transformers 不含 Higgs Audio V2 tokenizer,自己 vendored(内置拷贝)了一份 940 行的实现。现在钉定的 Transformers 5.6 已经官方内置同样的类,于是删掉 vendored 版、直接 import 上游类,只保留自带的 codec 配置 JSON 和现有权重加载路径,编解码接口一字不改。

知识点 —— vendoring(内置依赖): 把第三方代码拷进自己仓库,好处是不受上游版本约束、可控;坏处是要自己维护、跟不上上游修复。一旦上游提供了等价能力,及时"去 vendor"是减少维护负担的正确动作。PR 加了 meta-device 回归测试,钉住"确实绑定到了上游类""527 个 state-dict 键结构""帧率/码本数"等契约,确保切换零行为差异(WER 0.93%→0.86%,未劣化)。

可学到: 依赖管理的"内置↔上游"权衡;用回归测试锁住"重构前后接口/权重结构一致"。与 #1058、#1069 同属"给推理路径瘦身"主题。

#1069 · Remove unreachable implementation code   D · luojiaxuan · pull/1069

做了什么: 删掉仓库里"没有任何入口"的死代码——没有源码、测试、文档、配置字符串、装饰器或注册表引用它们。包括一个孤儿 environ 模块、一个没用到的 Fish 训练期 MoE loss、一个被取代的通用 cache-key 辅助函数,以及零散的 audio/VAD/Voxtral 小工具。非测试代码净删 266 行,不动任何流水线工厂或运行时接口。

知识点 —— 死代码 (dead code) 与可达性分析: "死代码"= 永远不会被执行到的代码。判断依据是"有没有任何入口点(entry point)可达"——源码调用、测试、配置引用、装饰器注册等都算入口。死代码不是无害的:它增加阅读负担、拖慢搜索、可能误导后人以为某功能还在用,还扩大安全/维护面。PR 的验证方式很扎实:删完跑 import 冒烟测试 + 编译检查 + 6 个 cache-key 单测,确认没误删活代码。

可学到: 定期做"可达性清理";删除前先证明"确无入口",删除后用冒烟/编译测试兜底。

#1072 · Align Qwen3-Omni gates with H100 calibration report   F · Hayden727 · pull/1072

做了什么: 更新 Qwen3-Omni 的 CI 质量门槛。原来的阈值是从旧 PR #1021 抄来的,和当前代码+环境跑出来的真实数据对不上。这个 PR 用"在当前 PR head 上、每个 stage 跑 5 次、取最差 (worst-of-5)"的严格校准数据,重设 18 个 stage 的参考值(涵盖 MMSU/VideoMME/VideoAMME 的准确率、速度、Talker WER/RTF 等),并移除了一个过时的 155s 硬超时上限。

知识点 —— CI 质量门 (quality gate) 与 worst-of-5 校准: 推理系统的 CI 不只测"能不能跑",还要测"准确率/速度有没有退化"。但 GPU 评测有波动,单次跑不可靠。"跑 5 次取最差" 是在给阈值设一个稳健下界——只要真实表现别比"最差的一次"还差就算通过,能显著降低偶发抖动导致的假失败 (flaky failure)。校准数据还标注了主机、镜像、commit、完整可复现路径。相关领域概念:RTF(Real-Time Factor,实时率,<1 表示比实时更快)、WER(词错误率)、UTMOS(音质 MOS 预测分)、apply_slack(在参考值上留出容差再作为门槛)。

可学到: 评测阈值要"从当前系统重新校准",而非从旧 PR 硬抄;用重复观测取稳健统计量对抗测量噪声;门槛值要可复现、可追溯。

今日术语速查表

Stage / 阶段 — 推理流水线里一段独立的工作单元(预处理/生成/声码),可各自指定进程与 GPU。
Scheduler / 调度器 — 每个阶段前的排队与批处理组件,决定请求怎么攒批、怎么进 GPU。
KV Cache — 自回归生成时缓存历史 token 的 Key/Value,推理主要显存消耗者;max_total_tokens 是其容量上限。
CUDA Graph — 把一串 GPU kernel 录制成图一次性回放,消除 launch 开销;有固定形状/范围限制,超出掉回 eager。
Vocoder / 声码器 — 把音频 codec token 解码成可听 PCM 波形的最后一站。
VQ / codec token — 向量量化把音频压成离散码,使语音能像文本 token 一样被 Transformer 建模;多码本表达更丰富细节。
CUDA MPS — 让多个进程的 kernel 真正并发共享一张 GPU 的 NVIDIA 机制。
DP / TP — 数据并行(多个完整副本各处理不同请求)vs 张量并行(一个模型切分到多卡协作)。
OMP_NUM_THREADS / intra-op 池 — 控制单算子多线程并行度;不收敛会导致多进程线程超额订阅。
Single-flight / LRU / 内容寻址缓存 — 并发同键只算一次 / 按最近最少使用淘汰 / 用内容哈希做键(天然去重免过期)。
TTFC / RTF / WER / UTMOS — 首帧时间 / 实时率 / 词错误率 / 音质 MOS 预测——语音服务的核心指标。
fail-closed / fail-loud — 出错时倒向安全侧(隔离资源)/ 出错时立刻大声报错(暴露 bug)。
