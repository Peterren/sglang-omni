PR #692性能优化Higgs TTS多阶段
[Higgs] Optimize Higgs async decode resolve output handling
作者 @Ratish1 · 状态 merged · 查看 PR ↗
💡 一句话：在 Higgs TTS 的"异步解码"路径里，把只用于输出处理的那块小张量从 GPU 上构建改为直接在 CPU 上构建，省掉一次无谓的 GPU 操作与隐式同步，端到端吞吐 +13~15%。

2. 它解决了什么问题

Higgs 是一个 多码本（multi-codebook）自回归 TTS 模型：它每"走一步"不是吐出 1 个 token，而是同时吐出多个 codebook 的离散音频码。为了把 GPU 算得快、又不让 CPU 拖后腿，SGLang-Omni 给它实现了一条 异步解码（async decode） 流水线——GPU 先把这一步算完并"抢跑"下一步，CPU 在后面慢一拍地把结果搬回主机做整理。

痛点在于：异步路径里那一步"滞后的整理"（resolve）其实只需要一个 CPU 上的小列表（拿来做输出处理、马上要 .tolist() 转成 Python list）。但旧代码不分场景，无论同步还是异步，都把这个结果张量建在 GPU 上。在异步整理阶段这就等于：先在 GPU 上 new 一个张量、再立刻拷回 CPU——做了一趟完全没必要的来回，还可能引入隐式的设备同步，打断 GPU 的"抢跑"节奏。

类比：厨房里大厨（GPU）已经把下一道菜的料备好抢先开火了；传菜员（CPU）只需要把上一道菜的"小票"念给收银台听。旧做法却让传菜员每次都先跑回厨房、在灶台上重抄一张小票再拿出来念——明明手里那张纸条（CPU 张量）就够用。这个 PR 就是让传菜员直接用手里的纸条。

3. 具体做了什么改动

核心文件只有一个：sglang_omni/models/higgs_tts/model_runner.py。改动的本质是给内部函数 _decode_collect_host 增加一个 keyword-only 参数 next_token_device，用它来决定"输出张量建在哪个设备上"。

(a) 给 collect 函数加上"目标设备"开关
def _decode_collect_host(
-     self, combined_cpu, result, requests
+     self, combined_cpu, result, requests, *, next_token_device: torch.device | None
 ) -> None:
     ...
-    result.next_token_ids = torch.tensor(
-        cb0_per_row, dtype=torch.long,
-        device=result.logits_output.next_token_logits.device,   # 永远在 GPU 上建
-    )
+    if next_token_device is None:                 # 异步 resolve：直接建 CPU 张量
+        result.next_token_ids = torch.tensor(cb0_per_row, dtype=torch.long)
+    else:                                         # 同步路径：建在 logits 所在的 GPU
+        result.next_token_ids = torch.tensor(
+            cb0_per_row, dtype=torch.long, device=next_token_device,
+        )

cb0_per_row 是一个普通的 Python list[int]——每个请求"第 0 号 codebook"这一步采到的 token。把它喂给 torch.tensor(...) 时，不指定 device 就默认建在 CPU，指定了才会触发一次 host→device 拷贝。

(b) 两个调用点分别传入不同的设备
# 同步 / CUDA-Graph 路径（_collect_step_outputs_cg）：结果要喂给"下一步"，必须留在 GPU
+    next_token_device=result.logits_output.next_token_logits.device,

# 异步 resolve 路径（post_decode_resolve）：只做输出处理，传 None → 建在 CPU
+    next_token_device=None,
(c) 为什么异步路径"丢掉 GPU 张量"是安全的？

关键在于异步路径的另一半（post_decode_launch，GPU 半边）早就已经把下一步要用的 codebook-0 token 在 GPU 上发布好了。下面是同仓库里 launch 阶段的真实注释（model_runner.py 第 84–94 行）：

# Set next_token_ids (cb0) from GPU state now, with NO host sync, so the
# AR input chain (next step's input_ids = this step's output_ids) is
# available at launch — the host collect (post_decode_resolve) lags by one step.
result.next_token_ids = (
    self.model._cg_codes_BN[:n_real, 0].clamp_min(0).to(torch.long).clone()
)

也就是说：在异步模式下，"驱动下一步自回归"的那份 GPU 张量已经在 launch 阶段产出了；滞后一步的 resolve 只负责"把这一步的结果整理成给用户看的输出"，它要的只是一个 CPU 列表。所以把它建在 CPU 上不仅不影响正确性，反而省掉了"建在 GPU 再拷回来"的浪费。PR 描述也明确：采样数学、logits、温度/top-p/top-k、停止条件全部未动，只动了"输出张量落在哪个设备"。

4. 涉及的知识点 （重点·小白展开）

▸ 多码本 TTS / RVQ codebook（多个"声音通道"一起预测）

它是什么：普通语言模型一步预测 1 个文字 token；而神经音频编解码器（如 RVQ，残差向量量化）会把一帧声音压成 多个并行的离散码（codebook 0、1、2…）。Higgs 这类模型一步要同时预测好几个 codebook 的码，再由声码器还原成波形。

为什么需要它：单个 codebook 表达力不够（音质差）；用多个 codebook 残差叠加，才能在低比特率下保留细节。代价是模型每步要处理"一摞"码而不是一个 token。

本 PR 里怎么用：cb0_per_row 取的就是每个请求"codebook-0"的码，它被用来当作下一步自回归的输入接力（input_ids）。代码里的 _cg_codes_BN 形状是 [Batch, Num_codebooks]，[:, 0] 就是取 0 号码本那一列。

▸ Model Runner（模型运行器）与多阶段流水线

它是什么：在 SGLang 体系里，Model Runner 是真正"把一个 batch 喂进模型、跑一次前向、收集输出"的执行单元。SGLang-Omni 把 TTS/Omni 拆成 preprocessing → AR engine → codec → vocoder 等多个阶段（stage），每个阶段可以跑在不同进程甚至不同 GPU 上。

为什么需要它：把"取数据 / 前向 / 采样 / 收集结果"封装在 runner 里，调度器（scheduler）只管"谁先跑、跑多大 batch"，职责清晰；多阶段拆分则让 CPU 预处理、GPU 推理、声码器解码能各自并行、互不阻塞。

本 PR 里怎么用：改的就是 HiggsTTSModelRunner 里"收集这一步输出"的逻辑。post_decode（同步）、post_decode_launch / post_decode_resolve（异步两半）都复用同一个 _decode_collect_host，所以加一个参数就能让两条路径各取所需。

▸ 异步解码 & Host↔Device 拷贝 / 隐式同步

它是什么：GPU 计算是异步排队的——CPU 把任务塞进 CUDA stream 就返回了，GPU 在后台慢慢算。"异步解码"利用这一点：GPU 先把下一步算起来（launch），CPU 在后面一步把上一步结果搬回主机（resolve）。Host→Device（H2D）/ Device→Host（D2H）拷贝就是 CPU 内存与显存之间搬数据。

为什么需要它：如果每一步都"算完→立刻把结果拷回 CPU→等拷完→再算下一步"，GPU 会频繁空等（这种等待叫隐式同步，比如 .item()、跨设备建张量都可能触发）。异步流水线把搬运藏到下一步计算的背后，GPU 几乎不停。

本 PR 里怎么用：旧代码在异步 resolve 里 device=GPU 地建张量，相当于在"本该轻量的整理步"又戳了一下 GPU、多搬一趟数据。改成在 CPU 上建，既匹配它"马上要 .tolist()"的用途，又避免打断 GPU 抢跑。Benchmark 显示吞吐 +13~15%、RTF 降约 6%，正是这种"去掉一次小同步"在高并发下被放大的效果。

▸ CUDA Graph 与 one-step lookahead（抢跑一步）

它是什么：CUDA Graph 把一串固定的 GPU kernel"录制"成一张图，之后整张图一次性重放，省掉每个 kernel 的启动开销（launch overhead）。代码里 _cg_* 前缀（cg = CUDA Graph）的缓冲区就是为重放预分配的固定内存。

为什么需要它：TTS 解码每步很小、kernel 很多，CPU 逐个发射 kernel 的开销会成为瓶颈。录成图重放后，CPU 只需"按一次播放键"。配合 lookahead（提前一步发起下一步），GPU 利用率进一步拉满。

本 PR 里怎么用：post_decode_launch 在重放图后不做 host 同步就直接发布 GPU 上的 cb0，让自回归链条不断；真正的主机整理滞后一步在 resolve 完成。本 PR 保证了这个滞后整理只产出 CPU 张量，不再回头碰 GPU。

5. 我能学到什么 / 延伸思考

  • "为正确性付出的代价应当只付一次"。同一份整理逻辑被同步/异步两条路复用，作者没有复制粘贴，而是用一个参数把"设备归属"这件唯一的差异抽出来——既复用又不牺牲性能。
  • 性能优化常常藏在"无害但多余"的细节里。这里没有改算法、没有上新 kernel，只是删掉一次"GPU 上建张量再拷回"的来回，就能在高并发下放大成两位数吞吐提升。
  • 区分"驱动计算的张量"和"给用户看的张量"。前者必须在 GPU、参与下一步；后者只是结果快照，放 CPU 最省。识别清楚数据的"用途"，才知道它该住在哪。
  • 延伸前置知识：CUDA stream / event 与异步执行模型、torch 的 pinned memory 与 non_blocking=True 拷贝、RVQ 神经音频编解码器、SGLang 的 scheduler↔model-runner 分工。
PR #691注意力后端LLaDA2-Uni多模态
[LLaDA2-Uni] Set default attention backend to flashinfer
作者 @btw616 · 状态 merged · 查看 PR ↗
💡 一句话：给多模态扩散语言模型 LLaDA2-Uni 显式钉死注意力后端为 flashinfer，修复"关掉 CUDA Graph 后注意力后端被自动选成不支持的那个"的连带 bug。

2. 它解决了什么问题

LLaDA2-Uni 是一个 多模态扩散语言模型（DLLM），能吃文本+图像、产出文本。它唯一被完整验证支持的注意力后端是 FlashInfer。但之前代码从没显式声明过这件事——它是"顺带"被保证的：早先默认开启 CUDA Graph，而开 CUDA Graph 时框架恰好会把后端选成 FlashInfer。换句话说，正确的后端是靠一个无关设置的副作用蒙对的。

最近有个改动默认关掉了 LLaDA2-Uni 的 CUDA Graph（这条实验性 DLLM 路径开图不稳定）。这一关，"顺带保证后端"的隐含链条就断了：框架的自动后端选择逻辑可能挑中一个 LLaDA2-Uni 根本不支持的后端，于是启动/推理直接出问题。本 PR 把这层隐式依赖变成显式声明。

类比：你家的灯一直亮着，是因为"开空调时顺手也把灯打开了"。某天有人说"空调太吵，默认别开了"，结果灯也黑了——因为灯从来没有自己的开关，一直蹭空调的。这个 PR 就是给灯装上专属开关，不再依赖空调。

3. 具体做了什么改动

仅一处，文件 sglang_omni/models/llada2_uni/stages.py。在构造 thinker 阶段的 server 参数时，把 overrides 字典从"只关 CUDA Graph"扩展为"显式指定 flashinfer + 关 CUDA Graph"：

-    overrides = {"disable_cuda_graph": True}
+    overrides: dict[str, Any] = {
+        "attention_backend": "flashinfer",
+        "disable_cuda_graph": True,
+    }
     overrides.update(server_args_overrides or {})

逻辑前后对比：

  • 改动前：只设 disable_cuda_graph=True。attention_backend 留空 → 交给框架自动推断 → 在"无 CUDA Graph"语境下可能选到非 FlashInfer 后端 → 崩。
  • 改动后：显式 attention_backend="flashinfer"，不再让框架猜。同时保留 overrides.update(...)，意味着用户若在配置里显式传了别的后端，仍可覆盖——默认值安全，但不剥夺高级用户的控制权。
设计细节：注意 overrides.update(server_args_overrides or {}) 放在最后一行执行，所以这里写的是默认值而非强制值——这是"安全默认 + 可覆盖"（sane default, still overridable）的经典写法。

4. 涉及的知识点 （重点·小白展开）

▸ Attention Backend（注意力后端）是什么

它是什么：Transformer 的核心计算是"注意力"（attention）。同一个数学公式，可以由不同的高性能实现来算——FlashInfer、FlashAttention、Triton、纯 PyTorch 等，这些可互换的实现就叫"注意力后端"。它们各自对硬件、数据布局、KV-cache 排布、是否支持变长/分页有不同要求。

为什么需要它：注意力是推理里最吃显存带宽和算力的部分。挑对后端能显著提速、省显存；但不是每个后端都支持每种模型/特性（比如某些采样模式、分页 KV、特定 mask）。所以"模型 ↔ 后端"是有兼容性约束的。

本 PR 里怎么用：LLaDA2-Uni 这条实验路径只在 FlashInfer 上被验证过。PR 把它钉死为默认后端，避免框架自动选择时挑到不兼容的实现。

▸ FlashInfer 简介

它是什么：FlashInfer 是一个面向 LLM 推理服务的高性能注意力/采样算子库，提供分页 KV-cache、各种 prefill/decode 注意力核、以及对 SGLang 这类引擎友好的接口。SGLang 把它作为重点支持的后端之一。

为什么需要它：它针对"服务场景"（变长序列、连续批处理、分页 KV）做了深度优化，比通用实现更适合高并发推理。

本 PR 里怎么用：就是被显式指定的那个值 "flashinfer"。文档配套也补了一句："thinker 在这条实验 DLLM 路径下默认关闭 CUDA graph"。

▸ CUDA Graph 为什么会"顺带"决定后端

它是什么：前面讲过，CUDA Graph 把固定 kernel 序列录制后整体重放。但要能录制，所有 kernel 形状必须固定、且后端要支持"图捕获（graph capture）"模式。框架在"决定开图"时，往往会连带把注意力后端切到支持图捕获的那个（即 FlashInfer）。

为什么需要它：这正是隐式耦合的来源——两个本应独立的配置（是否开图、用哪个后端）被框架的自动逻辑绑在了一起。一旦改了其中一个，另一个的取值就跟着变。

本 PR 里怎么用：作者识别出这条隐式链，主动解耦：无论开不开图，后端都明确是 FlashInfer。这就是"消除隐式依赖、改为显式契约"的工程思想。

▸ 扩散语言模型 DLLM & thinker 阶段

它是什么：主流 LLM 是自回归（一个一个 token 往外吐）；扩散语言模型（Diffusion LLM）则像图像扩散那样，从"全是 mask 的占位"出发，分多轮并行去噪/填空，逐步把整段文本确定下来。LLaDA2 系列就属于这类。"thinker"是 Omni 流水线里负责"理解+生成文本表征"的核心阶段。

为什么需要它：扩散式生成在某些场景能更好地做全局规划、并行填充，是当前的研究热点。但它的调度/缓存模式和自回归不同，对引擎适配提出新要求（见下一张卡片 #670）。

本 PR 里怎么用：改的就是 create_sglang_dllm_thinker_executor_from_config——给 DLLM thinker 阶段配置后端。它和 #670 修的是同一个模型，一个修后端、一个修调度器，互为补充。

5. 我能学到什么 / 延伸思考

  • "靠副作用蒙对"是技术债。当一个正确行为依赖于另一个无关配置的副作用时，它随时会因为那个配置的变动而无声崩坏。把隐式依赖写成显式声明，是消除这类脆弱性的标准手法。
  • 安全默认 + 仍可覆盖。把默认值放进 overrides 字典、再让用户配置 update 在其后，是兼顾"开箱即用"和"高级可控"的小而美的写法。
  • 一行代码的价值取决于它背后的因果链。这个 PR 只加了一行，但价值在于作者讲清了"关图 → 自动选后端 → 选错 → 崩"的完整链路。读 PR 要读因果，不要只读 diff。
  • 延伸前置知识：FlashInfer/FlashAttention 的差异、注意力后端与 KV-cache 布局的兼容关系、扩散语言模型的采样过程、SGLang server_args 的后端自动选择逻辑。

2. 它解决了什么问题

SGLang-Omni 把 sglang 作为底层推理引擎依赖。但开发镜像里跑的是 sglang 0.0.0.dev1（比锁定的 0.5.8 更新），上游把 PrefillAdder 的接口改了：

  • 它的 __init__ 不再接受 dllm_config 参数；
  • 原来的辅助属性 PrefillAdder.dllm_staging_reqs 被删除了。

于是 LLaDA2-Uni 旧的扩散调度器路径在"还没碰到模型"时就抛异常——也就是说，/v1/chat/completions 对这个模型根本调不通。本 PR 把调度器改写到新 API 上，让文本对话请求能正常返回（已在 dev 镜像上实测：返回连贯文本、finish_reason: stop、HTTP 200）。

上游漂移（upstream drift）的典型案例：你依赖的库悄悄改了内部 API，你的适配层就断了。这类问题不在你的代码里、却让你的功能挂掉，排查尤其费神。

3. 具体做了什么改动（核心：scheduling/dllm_scheduler.py）

(a) chunked-prefill 尺寸来源更稳健
+    self._chunked_prefill_size = (
+        getattr(dllm_config, "block_size", None) or server_args.chunked_prefill_size
+    )

扩散模型按"块（block）"去噪，块大小更贴合它的语义；拿不到就回退到 server 通用值。getattr(..., None) or ... 是 Python 里"优先取 A，没有就取 B"的惯用写法。

(b) 构造 PrefillAdder 时不再传被删掉的参数
     adder = PrefillAdder(

4. 涉及的知识点 （重点·小白展开）

▸ Scheduler（调度器）与 PrefillAdder

它是什么：调度器是推理引擎的"交通指挥"：决定当前这一拍把哪些请求、组成多大 batch 喂给 GPU。PrefillAdder 是 SGLang 里负责"prefill 阶段往 batch 里塞请求"的组件——它会按显存预算、token 上限、KV-cache 容量来决定能塞几个、塞多少。

为什么需要它：推理是多请求共享一块 GPU。塞太多会 OOM，塞太少浪费算力。PrefillAdder 就是把"能塞的请求"精确算出来的那段逻辑。

本 PR 里怎么用：扩散调度器复用 PrefillAdder 来组 batch；上游改了它的签名（去掉 dllm_config、删了 dllm_staging_reqs），本 PR 把调用方式迁移到新接口，并用 prefill_max_requests=1 显式表达"这条实验路径一次只跑一个请求"。

▸ Prefill vs Decode / Chunked Prefill

它是什么：自回归推理分两段：Prefill（一次性并行处理整段输入 prompt，建好 KV-cache）和 Decode（之后一个一个往外吐 token）。Chunked Prefill：当 prompt 很长时，把 prefill 切成若干小块分批做，避免一次占满显存、也让长 prompt 不至于饿死其他请求。

为什么需要它：长输入的 prefill 是显存/算力尖峰；分块能平滑这个尖峰，提升整体并发与公平性。

本 PR 里怎么用：(a) 处优先用扩散模型自己的 block_size 作为 chunk 尺寸——因为 DLLM 本来就是按块去噪的，块大小语义最贴合。

▸ 扩散调度的特殊性：为什么要"反复调度同一请求"

它是什么：自回归请求每步产出一个 token、走很多步直到 EOS；扩散请求则是对整个序列做多轮去噪，每一轮都要再进一次模型，直到所有位置都"定稿"。

为什么需要它：因为同一个请求要被调度器拉起好多轮，调度器必须自己记住"哪些请求还在去噪中"（staging 队列），不能 prefill 一次就丢掉。

本 PR 里怎么用：(c) 处用 can_run_list 重建这个 staging 队列，正是为了在新 API 下继续维持"扩散请求的多轮生命周期"。

4. 涉及的知识点 （重点·小白展开）

▸ PCM / 波形 / 容器 / 编解码器（codec）四件套

它是什么：PCM/波形是声音的"原始数字采样"（一串浮点数）；编解码器 codec（mp3/aac/opus…）是把波形压缩成更小字节流的算法；容器（ogg/adts/mp3 文件）是把压缩后的数据流装进一个带头部、能被播放器识别的"盒子"。

为什么需要它：裸 PCM 体积大、不带元信息；压缩后小很多，但必须装进容器、声明采样率/声道，播放器才认。

本 PR 里怎么用：配置表里 container（盒子）和 codecs（压缩算法）是分开指定的——比如 opus 用 ogg 容器装 libopus 编码的数据。

▸ PyAV vs pydub：进程内绑定 vs 外部命令

它是什么：pydub 是高层封装，运行时去 shell 调系统的 ffmpeg 可执行文件；PyAV 是 FFmpeg 库（libav*）的 Python 原生绑定，在进程内直接调 C 库，不依赖外部命令、不开子进程。

为什么需要它：外部命令依赖环境装没装、版本对不对、PATH 找不找得到，部署脆弱且报错难定位；进程内绑定可控、可捕获异常、可精确控制采样率/帧格式。

本 PR 里怎么用：把首选编码路径换成 PyAV，同时保留 pydub 作为降级——既拿到稳定性，又不破坏老环境的兼容。

▸ 采样率 & 重采样（resampling）

它是什么：采样率是每秒采多少个声音样本（24000Hz = 每秒 2.4 万点）。重采样是把一段音频从一个采样率换算到另一个（点数随之改变）。

为什么需要它：每种压缩编码器只支持特定采样率集合。模型输出 24kHz 多半 OK，但若上游给了 44.1k/12345 这种值，就得先换到编码器认的最近值。

本 PR 里怎么用：_resample_linear 用线性插值做轻量换算；valid_rates 白名单 + min(..., key=abs(r-sr)) 找最近合法采样率。

5. 我能学到什么 / 延伸思考

  • "优雅降级"是服务化代码的底线。三层兜底（PyAV → pydub → WAV）保证用户永远拿得到能播的音频，哪怕画质打折也不返回错误——这是面向最终用户接口的健壮性思维。
  • 把外部进程依赖换成进程内库，是提升可部署性的常见升级。少一个"系统得装某个二进制"的前提，容器就更省心、错误更可捕获。
  • 用魔数测二进制格式。不解码、只看文件头几个字节，就能稳定地断言"编码确实成功且格式正确"，比"能不能播放"更适合自动化测试。
  • 延伸前置知识：数字音频基础（采样率/位深/声道）、FFmpeg 的容器与编码器关系、planar vs interleaved 音频帧格式、编码器内部缓冲与 flush、Nyquist 采样定理与重采样滤波。

4. 涉及的知识点 （重点·小白展开）

▸ 并发 / 扇出（concurrency, fan-out）与连续批处理

它是什么：并发指同时在途的请求数。推理引擎用连续批处理（continuous batching）把多个在途请求动态拼成一个 batch 喂 GPU——谁完成谁退出、空位立刻补新请求，GPU 始终满载。扇出就是客户端一次性放出多少并发请求。

为什么需要它：高并发能把 GPU 喂饱、提升稳态吞吐；但在小负载上，高并发主要制造排队和尾延迟，并不提速。所以"最优并发"和工作负载大小强相关。

本 PR 里怎么用：新脚本就是用来"扫描"不同并发档位、量化它们对吞吐/延迟/WER 的影响，从而为"小门禁用并发 2、大评测用并发 32"提供数据支撑。

▸ WER / RTF / 延迟分位（p95/p99）这些指标怎么读

它是什么：WER（Word Error Rate，词错率）衡量转写质量，越低越好；RTF（Real-Time Factor，实时率）= 处理耗时 / 音频时长，<1 表示比实时还快；p95/p99 延迟是"95%/99% 的请求都比这个值快"，反映尾部体验。

为什么需要它：只看均值会掩盖长尾。p95/p99 才暴露"最慢的那一小撮"；RTF 让不同时长音频可比；WER 衡量"快了但准不准"。

本 PR 里怎么用：基准脚本对每个并发档位都报全套指标，并跨多次重复取 mean/min/max，给出可复现、抗噪声的"并发-扩展曲线"。

▸ CI 门禁 & 阈值校准（threshold calibration）

它是什么：CI 门禁是合并前自动跑的"红线测试"，不达标就拦下。阈值校准是把这些红线设到"刚好能放过正常波动、又能挡住真实退化"的位置。

为什么需要它：阈值太松 = 退化漏网；太紧 = 正常抖动也误报。关键是阈值必须基于"门禁真正运行的那个工作负载"来定——这正是本 PR 修的 bug。

本 PR 里怎么用：把阈值从并发=32 的旧值改到并发=2 的实测值，并让默认并发、本地运行、tune-ci-thresholds 工具三者对齐到同一负载。

5. 我能学到什么 / 延伸思考

  • "测试要测你真正在跑的东西"。当门禁的阈值和它实际运行的负载脱节，门禁就退化成橡皮图章。校准必须基于真实工作负载。
  • 让测试与基准同源，消灭"度量漂移"。把转写/打分逻辑抽成共享函数，CI = 跑基准 + 套阈值，从根上避免"两套算法各算各的"。
  • 性能直觉要用数据验证，而非想当然。"并发越高越快"在小负载上是错的；本 PR 用扫描数据证明了这点，并据此分场景选并发。
  • 延伸前置知识：continuous batching 与 PagedAttention、吞吐 vs 延迟的权衡、长尾分位统计、ASR 评测流程（归一化 + jiwer 计算 WER）、负载测试方法论。
