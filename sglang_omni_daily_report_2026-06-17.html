#816 [Higgs] Fuse top-k/top-p sampling with sgl_kernel renorm
MERGED多模态·TTS性能优化作者 @BBuf · 合并于 2026-06-17 08:22 UTC · 查看 PR
 ▸

1 PR 概览

  • 标题：[Higgs] Fuse top-k/top-p sampling with sgl_kernel renorm
  • 编号 / 作者 / 状态：#816 · @BBuf · 已合并 (merged)
  • 链接：github.com/sgl-project/sglang-omni/pull/816
  • 一句话：把 Higgs Audio v3 这个 TTS 模型在“逐步生成音频 token”时所用的 top-k/top-p 采样，从“全词表排序 + 掩码”的慢实现，换成 sgl_kernel 里写好的两个融合 CUDA kernel（top_k_renorm_prob / top_p_renorm_prob），在数值几乎完全等价的前提下把每步解码 GPU 时间降了约 1.7%。

2 它解决了什么问题

Higgs 是一个文本转语音（TTS）模型。它生成音频的方式不是直接吐波形，而是像 LLM 吐文字 token 一样，一步一步地吐“音频 token”（codebook token）。每一步都要从一个概率分布里“抽签”决定下一个 token，这个抽签过程就叫采样（sampling）。

作者用性能分析工具（profiler）在 H100 上跑 Higgs 解码，发现一个痛点：每一个解码步，采样器都在做一条“全词表排序”的重活——

  • 对整个词表（vocab，这里 V=1026，而且有 8 个 codebook 同时采样）做 torch.sort 完整排序；
  • 再 topk(K_MAX) 取前 K 大；
  • 再 cumsum（累加和）+ scatter（散射写回）来做 top-p 的掩码。

profiler 看到的具体 CUDA kernel 是 radixSortKVInPlace、sbtopk::gatherTopK、softmax_warp_forward 一长串。问题在于：词表很小（才 1026），但这些 kernel 一个接一个地启动，每个都很短，GPU 大部分时间不是在算，而是在“等下一个 kernel 被启动”——这叫 launch-bound（启动开销受限）。就像让一个大厨切一片黄瓜，结果光是“拿刀、放刀、再拿刀”的动作比切本身还久。

排序是为了找 “前 K 大 / 累计概率到 p”，但其实你根本不需要把所有元素排好序，只需要知道“阈值在哪、谁该保留”。为此付出一次 O(V log V) 的全排序，是杀鸡用牛刀。融合 kernel 直接在一个 kernel 内部用更聪明的算法（partial selection + 重归一化）算出结果，省掉了排序和多次 kernel 往返。

3 具体做了什么改动

唯一改动的文件：sglang_omni/models/higgs_tts/sampler.py，函数 _sample_independent_batched。下面把改动前和改动后对照讲清楚。

▼ 改动前（旧逻辑：排序 + 掩码）

# top-k：对 logits 做 topk，取第 k 大的值作为阈值 kth，
# 小于阈值的位置全部置为 -inf（即丢弃）
if top_k_buf is not None:
    top_vals = logits.topk(K_MAX, dim=-1).values
    k_idx = top_k_buf.view(B,1,1).expand(-1,N,1).clamp(min=1,max=K_MAX) - 1
    kth = top_vals.gather(-1, k_idx)
    logits = torch.where(logits < kth, float("-inf"), logits)

# top-p（核采样）：先全量排序，算累计概率 cumsum，
# 累计超过 p 的尾部全部丢弃，再 scatter 写回原位置
if top_p is not None:
    sorted_logits, sorted_indices = torch.sort(logits, descending=True, dim=-1)
    cum_probs = sorted_logits.softmax(dim=-1).cumsum(dim=-1)
    remove = cum_probs > top_p.view(B,1,1)
    remove[..., 1:] = remove[..., :-1].clone()   # 右移一位，保留刚好越过 p 的那个 token
    remove[..., 0] = False
    scatter = torch.zeros_like(remove)
    scatter.scatter_(-1, sorted_indices, remove)        # 把“该删”的标记散射回原始顺序
    logits = torch.where(scatter, float("-inf"), logits)

probs = logits.softmax(dim=-1)
codes_flat = probs.reshape(B*N, V).multinomial(num_samples=1).squeeze(-1)

▼ 改动后（新逻辑：先 softmax 成概率，再调融合 kernel 重归一化）

# 关键：先把 logits 变成概率分布，强制 contiguous + fp32（kernel 硬性要求）
probs = logits.float().softmax(dim=-1).reshape(B*N, V).contiguous()

if top_k_buf is not None:
    tk = top_k_buf.view(B,1).expand(B,N).reshape(B*N) \
            .clamp(min=1,max=V).to(torch.int32).contiguous()
    probs = _fused_top_k_renorm(probs, tk)        # 一个 kernel 搞定 top-k + 重归一化

if top_p is not None:
    tp = top_p.view(B,1).expand(B,N).reshape(B*N).to(torch.float32).contiguous()
    probs = _fused_top_p_renorm(probs, tp)        # 一个 kernel 搞定 top-p + 重归一化

codes_flat = probs.multinomial(num_samples=1).squeeze(-1)   # 抽签（保持不变）

逐块解读两者的差别：

  • 旧逻辑在 logits 空间操作：用排序找阈值、用 -inf 掩码、最后才 softmax。中间产生 sorted_logits、sorted_indices、scatter 等一堆临时大张量，且每个 PyTorch 算子都是一次独立 kernel 启动。
  • 新逻辑在 probability 空间操作：先 softmax 成概率，然后直接交给 _fused_top_k_renorm / _fused_top_p_renorm。这两个是 sgl_kernel（背后是 FlashInfer）提供的 CUDA kernel，它们内部一次性完成“筛选 + 把剩下的概率重新归一化到和为 1”，不再需要 PyTorch 层面的排序和散射。
  • 为什么叫 renorm（重归一化）：当你把一部分 token 的概率砍掉后，剩下概率之和不再是 1，必须重新除以新的总和，multinomial 抽签才正确。融合 kernel 把“砍 + 归一化”合并成了一步。
  • 三个工程细节（很关键，新手最容易踩坑）：
    • .contiguous() 和 .float()：FlashInfer 的 renorm kernel 硬性要求输入是连续内存的 fp32 张量。PR 描述里特别强调“非连续张量会静默地破坏结果”——即不报错但算错，这是最阴险的 bug，所以必须显式 contiguous。
    • .to(torch.int32)：top_k 参数要转成 int32，因为 kernel 接口签名是这么定义的。
    • greedy / argmax 短路和 CUDA graph 安全的 multinomial 保持原样没动——作者只换了中间这段，最大程度减少风险面。

4 涉及的知识点（扫盲精讲）

框架 / 领域
TTS 的自回归 token 生成 & codebook（码本）

是什么：现代神经 TTS（如 Higgs、MOSS）不直接生成波形，而是先用一个“音频编解码器（neural audio codec，如 EnCodec/SNAC 类）”把声音离散化成一串整数 token（就像把图片压成 JPEG 的系数）。模型自回归地预测这些 token，最后再用解码器（vocoder）还原成 24kHz 波形。codebook 就是这些离散 token 的“字典”；Higgs 用 8 个 codebook 并行（multi-codebook），所以一步要同时为 8 路各采一个 token。

为什么需要：把连续音频离散化后，就能完全复用 LLM 那一套“预测下一个 token”的成熟基础设施（KV cache、batching、采样）。

本 PR 里怎么用：被优化的 _sample_independent_batched 就是“为这 8 路 codebook 各自独立采样”的函数，形状 B×N×V 里 N 就是 codebook 数。

采样算法
top-k / top-p（核采样）与温度（temperature）

是什么：模型每步输出一个对全词表的打分（logits）。直接按概率抽签容易抽到长尾垃圾 token；于是用三道“质量闸门”：temperature 调节分布陡峭程度（越低越保守）；top-k 只保留概率最高的 k 个候选；top-p（nucleus） 保留“累计概率达到 p”的最小候选集。

为什么需要：在 TTS 里直接关系到音质——采样太“野”会爆音、跑调；太“死”会机械、单调。

本 PR 里怎么用：算法语义完全不变，只是把实现从“排序版”换成“融合 kernel 版”。作者特别说明：唯一理论差异在 cumsum == top_p 的精确边界上，旧代码用 cum > p 再右移，新 kernel 用标准 cum >= p 约定；对连续 logits 这是“测度为零”事件（几乎不可能精确相等），实测 225 种配置里从未触发。

性能工程
Kernel 融合（fusion）与 launch-bound

是什么：GPU 每执行一个算子就要“启动一个 kernel”，启动本身有固定开销（几微秒）。当单个算子的计算量很小，启动开销就会主导总时间——这就是 launch-bound。Kernel 融合是把多个小算子合并成一个大 kernel，用一次启动完成多步计算。

为什么需要：Higgs 的词表只有 1026，每个排序/掩码算子算得飞快但启动一堆，典型的“启动比计算贵”。

本 PR 里怎么用：把 sort+topk+cumsum+scatter+softmax 这一长串，压成 softmax + 1~2 个 renorm kernel。profiler 显示采样 kernel 块从 ~64µs/步 降到 ~32µs/步（砍半）。

就像点外卖：与其分别下 5 个单（5 次配送费），不如合并成 1 单。计算（菜）没变，省的是“配送费”（kernel 启动）。
推理基建
sgl_kernel / FlashInfer 与 CUDA Graph

是什么：sgl_kernel 是 SGLang 自带的高性能算子库，底层很多来自 FlashInfer（专为 LLM 推理写的 CUDA kernel 集合）。CUDA Graph 则是把一串固定的 GPU 操作“录制”成一张图，之后整图一次性重放，进一步消除 CPU 端逐 kernel 启动的开销。

为什么需要：解码阶段每步操作高度重复，CUDA Graph 能把上百次 launch 压成一次重放，是 LLM/TTS 解码提速的标配。

本 PR 里怎么用：作者验证了替换后“CUDA graph 捕获不受影响、端到端仍输出有效 24kHz 语音、无 graph break”。所谓 graph break 指引入了不能被图捕获的动态操作（如数据相关的控制流），会迫使图断开、退回慢路径——所以保持 multinomial 那段“graph-safe”很重要。

  • 等价替换 + 严格验证：用 225 种 (B × temperature × top_k × top_p) 组合做扫描，证明 max prob diff ~5e-7、保留集合完全一致，把“性能优化引入正确性回归”的风险压到最低。这是工业级优化 PR 的范式。

  • 魔鬼在细节：contiguous + fp32 这种“不满足会静默算错”的隐性契约，必须在代码和注释里显式处理。

#810 [MOSS] Tune MOSS Local colocated AR memory budgeting
MERGED多阶段流水线显存管理作者 @Ratish1 · 合并于 2026-06-16 15:56 UTC · 查看 PR
 ▸

1 PR 概览

  • 编号 / 作者 / 状态：#810 · @Ratish1 · 已合并 (merged)
  • 链接：github.com/sgl-project/sglang-omni/pull/810
  • 一句话：MOSS Local 这个 TTS 流水线在单卡上把“预处理 + 自回归(AR) + 声码器(vocoder)”三个阶段挤在同一张 GPU 上跑（colocated，共置）。本 PR 给中间最吃显存的 AR 阶段一套更精确、显式的显存预算方案——用“进程级显存核算”代替 SGLang 默认的“整卡空闲显存”估算，并预留一小块给后面加载的 codec，端到端吞吐 +8.9%、WER 还略降。

2 它解决了什么问题

先理解多阶段流水线（multi-stage pipeline）：MOSS Local 把一次 TTS 请求拆成三段串行处理——

  • preprocessing（预处理）：处理文本、参考音色等，会加载一个 codec（编解码器）。
  • tts_engine / AR（自回归）：核心大模型，逐 token 生成音频 token，最吃显存（要放权重 + KV cache）。
  • vocoder（声码器）：把 token 还原成波形，又要加载一个 codec 实例。

在单卡部署里，这三段共用同一张 GPU（colocated）。痛点来了：SGLang 引擎启动时要决定“我能用多少显存放 KV cache”，它默认的做法是看整张卡当前还剩多少空闲显存，按比例 mem_fraction_static 切一块。但在共置场景下，这个“整卡空闲”是不准的——预处理的 codec 已经占了一块、后面 vocoder 的 codec 还没加载但迟早要占。如果 AR 阶段贪心地按“当前空闲”吃满，等 vocoder 来加载 codec 时就OOM（显存爆掉）。

旧代码的做法是“拍脑袋”给个静态数：mem_fraction_static = 0.6（多卡）或 0.5（单卡）——保守、浪费，而且没有理论依据。本 PR 要做的是：把这套预算显式化、结构化、并接入更准的“进程级显存核算”。

三个室友合租一间房（一张 GPU）。老办法是老二（AR）进门看“现在地上还空多少”就铺多大的床，结果老三（vocoder）回来发现没地方睡了（OOM）。新办法是事先按合同分配：总共用 90% 的房间，给老三的家具预留 5%，老二自己实际用 85%——并且用“房产证上的精确面积（进程级核算）”而不是“目测空地”。

(B) stages.py — 新增预算计算函数 _apply_colocated_ar_memory_budget

def _apply_colocated_ar_memory_budget(overrides, *, total_gpu_memory_fraction, codec_mem_reserve):
    if total_gpu_memory_fraction is None:           # 分卡/旧路：不参与，返回空预算
        return _ArMemoryBudget(None, 0.0)
    if not 0.0 <= codec_mem_reserve < 1.0:
        raise ValueError("codec_mem_reserve must be in [0, 1)")

    # 有效预算 = 总预算 - 给 codec 的预留。0.90 - 0.05 = 0.85
    effective = round(total_gpu_memory_fraction - codec_mem_reserve, 3)
    if effective < 0.1:                          # 安全下限，预留太多会让 AR 没法跑
        raise ValueError("... below the safe floor 0.1 ...")

    # 若用户显式给了 mem_fraction_static，尊重它，但不许超过总预算
    explicit = overrides.get("mem_fraction_static")
    if explicit is not None:
        if explicit > total_gpu_memory_fraction:
            raise ValueError("... cannot exceed total_gpu_memory_fraction ...")
        effective = explicit
        applied_reserve = round(total_gpu_memory_fraction - effective, 3)
    else:
        overrides["mem_fraction_static"] = effective    # 默认就写回 0.85
    return _ArMemoryBudget(effective, applied_reserve)

解读三层逻辑：① 算出有效预算 0.85；② 把它透传给 create_sglang_infrastructure，让 SGLang 在“进程级显存”可用时按这个比例 profile；③ 如果环境拿不到进程级数据（比如某些容器/驱动），优雅降级回上游默认估算，绝不在不确定时硬上一个危险预算。这套“尝试更优 → 不行就安全回退”的写法非常值得学。

4 涉及的知识点（扫盲精讲）

系统架构
多阶段流水线 colocated（共置）vs split（分卡）

是什么：一个 Omni/TTS 系统由多个模型阶段组成。共置=全部塞进一张 GPU（省卡、省成本，但显存紧张、互相争用）；分卡=把不同阶段放到不同 GPU（宽裕但费卡）。

为什么需要：生产部署要在“成本”和“稳定性”间权衡。单卡共置最便宜，但显存预算稍有不慎就 OOM——正是本 PR 要根治的。

本 PR 里怎么用：用一个 colocated 布尔，让同一份 stage 工厂同时支持两种拓扑，并只给共置路打上更精细的预算。

推理基建（核心）
KV cache 与 mem_fraction_static / 显存 profiling

是什么：自回归生成时，每个已生成 token 的 Key/Value 向量要缓存起来供后续步骤注意力复用，这就是 KV cache，它随序列变长而增大，是推理显存的大头。SGLang 启动时会做一次“显存 profiling”：先加载模型权重，剩下的显存按 mem_fraction_static 预先静态分配成一个大 KV cache 池（避免运行时反复申请/碎片化）。

为什么需要：预分配一个固定池能让显存可控、避免运行中 OOM，也利于做 continuous batching。但“按整卡空闲算”在共置场景会高估可用量。

本 PR 里怎么用：把“整卡空闲”估算换成“进程级核算（只算本进程实际占用）+ 显式预留 codec”，让 KV cache 池的大小既不浪费也不踩 OOM。

显存核算
进程级显存核算（process-scoped accounting）vs 整卡空闲

是什么：“整卡空闲”= 看 nvidia-smi 那种全卡剩余；“进程级”= 精确知道当前这个 Python 进程自己占了多少。共置时同卡上别的阶段也在占显存，只有进程级数才能让 AR 准确知道“我还能安全吃多少”。

本 PR 里怎么用：通过 get_process_gpu_memory_bytes(gpu_id) 判断该机制是否可用；可用就走精确预算，不可用就 warning + 回退。

设计模式
不可变值对象 + 显式校验 + 优雅降级

是什么：用 @dataclass(frozen=True) 的 _ArMemoryBudget 把“算出来的预算结果”打包成一个只读对象传递；所有边界（reserve∈[0,1)、effective≥0.1、explicit≤total）都显式 raise ValueError 校验；环境不支持时不崩溃而是降级。

为什么需要：显存预算错了会直接 OOM 崩服务，属于“失败代价很高”的配置，必须把非法值在启动期、配置期就拦下来，而不是运行到一半才炸。

领域概念
Qwen3-Omni 与“全模态（any-to-any）”服务

是什么：Omni 模型能接收多种模态（文+图+音）并生成多种模态（文+语音）。在 SGLang-Omni 里，它常是一个多阶段系统：thinker（理解+生成文本）→ talker（把文本转成语音 token）→ codec/vocoder（出波形）。

服务压测
闭环(closed-loop) vs 开环(open-loop)、并发 vs 速率

是什么：开环按固定 QPS 发请求（模拟真实用户到达，关心“能否扛住 X QPS”）；闭环维持固定在飞并发数（关心“满负荷下的极限吞吐/扩展性”）。

为什么需要：要刻画“并发扩展性曲线”，闭环最直接——逐档抬高并发，看吞吐是否线性增长、延迟何时拐点爆炸。

本 PR 里怎么用：request_rate=inf + max_concurrency=count = 闭环；扫 1/2/4/8/16/32 得到扩展性曲线。

rollout
同提示多路 rollout 与 KV / 前缀缓存

是什么：rollout 指同一输入并发跑多条生成（RL 训练采样、best-of-n 都靠它）。同提示意味着共享前缀，理论上可被 SGLang 的 RadixAttention / 前缀缓存复用，省掉重复的 prefill 计算。

本 PR 里怎么用：_duplicate_prompt 造出 N 份同提示并打同一 user/rollout_group_id，专门压这种流量。

是什么：均值会骗人，p95/p99 尾延迟才反映“最差那批用户的体验”；req/s 是吞吐，tok/s 是生成速度。profiler 事件则把端到端时间拆成“排队/prefill/decode”各段，定位瓶颈。

本 PR 里怎么用：每档都 compute_speed_metrics + print_speed_summary 输出这些数，结果连同 health/profile 存成 rollout_stress_results.json 供前后对比。

asyncio + aiohttp 做高并发客户端

是什么：压测客户端要同时维持几十路在飞请求，用线程会很重；asyncio 协程 + aiohttp 异步 HTTP 能用单线程高效管理大量并发连接（IO 等待时切去做别的）。
