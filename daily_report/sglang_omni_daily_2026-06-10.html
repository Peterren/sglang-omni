▶ #687 · 让 Ming-Omni 的图像编码器支持张量并行，并能从命令行开启
作者 @edwingao28 · 状态 ✓ Merged（2026-06-09 11:44 UTC）· 查看 PR →

1. PR 概览

  • 标题：[Feat] Enable Ming-Omni image_encoder tensor parallelism expose via CLI
  • 编号：#687　作者：@edwingao28（COLLABORATOR）　状态：已合并 merged
  • 链接：github.com/sgl-project/sglang-omni/pull/687

一句话概括：图像编码器（image_encoder）这个阶段在底层早就实现了张量并行（把一个大的视觉模型切到多张 GPU 上跑），但之前两个地方挡着用户：配置校验会直接拒绝、命令行也没有对应开关。这个 PR 把这两道「闸门」打开，让用户能用一行 sgl-omni serve 命令就把视觉编码器分到多卡上。

2. 它解决了什么问题

这里有一个非常典型的「能力已经造好，但没接上水管」的矛盾。背景是这样的：

  • 底层已经能跑。在更早的 PR #553 中，image_encoder.py 里的 _init_sglang_tp 方法已经实现了把视觉 Transformer（ViT）按张量并行切分到多个 rank（进程/卡）上。也就是说，引擎本身支持图像编码器的 TP。
  • 痛点 ①：校验环节一刀切地拒绝。在 PR #513 引入的「阶段 TP 白名单」validate_stage_tp_support 里，只允许 thinker（语言模型主干）开 TP。任何人想给 image_encoder 设 TP>1，都会在配置校验阶段被直接报错："Stage 'image_encoder' does not support TP in Ming-Omni V1"。这条报错和底层运行时的真实能力自相矛盾——明明跑得动，校验却说不行。
  • 痛点 ②：命令行根本没有开关。sgl-omni serve 给 thinker 准备了 --thinker-tp-size / --thinker-gpus，但视觉编码器一个 flag 都没有。用户要开视觉编码器 TP，唯一的办法是手写一份 YAML pipeline 配置文件——门槛极高、易错。
🍔 通俗类比

想象一家餐厅后厨已经装好了第二台烤箱（运行时支持 TP），但前台点单系统里既没有「双烤箱」这个选项（CLI 没 flag），而且就算你硬塞一张写着「用两台烤箱」的纸条进去，收银员还会照着一本过期的规则手册告诉你「本店只有一台烤箱，不接受这个要求」（校验白名单拒绝）。这个 PR 干的事就是：① 更新规则手册，② 在点单系统加上「双烤箱」按钮。

3. 具体做了什么改动

改动集中在 两个生产代码文件 + 三个测试文件。我们一个一个看。

改动 A：放开校验白名单 —— models/ming_omni/tp_utils.py

这是最核心、也最小的一处改动。原来有一个只含 thinker 的「集合（frozenset）」，校验函数拿用户请求的阶段名去这个集合里查，查不到就报错。

sglang_omni/models/ming_omni/tp_utils.py
- _MING_AR_TP_STAGES = frozenset({"thinker"})
+ _MING_TP_STAGES = frozenset({"thinker", "image_encoder"})

  def validate_stage_tp_support(*, stage_name: str, tp_size: int) -> None:
      if tp_size < 1:
          raise ValueError(f"Stage {stage_name!r}: tp_size must be >= 1, got {tp_size}")

-     if stage_name not in _MING_AR_TP_STAGES:
+     if stage_name not in _MING_TP_STAGES:

-             "Only the thinker stage is currently SGLang-TP aware."

逐行解释：

  • 变量改名 _MING_AR_TP_STAGES → _MING_TP_STAGES：原名里的 AR 是 "AutoRegressive（自回归）" 的缩写，暗示「只有自回归语言模型才支持 TP」。现在视觉编码器（非自回归）也支持了，所以把 AR 去掉，名字更诚实。这是一个很好的「命名跟着语义走」的习惯。
  • if tp_size == 1: return：TP=1 表示「只用一张卡、不切分」，对任何阶段都无意义可言，所以提前放行——这是一条快捷通道（early return），避免把 TP=1 也卡在白名单外。
  • 把 image_encoder 加进集合后，校验就和底层运行时的真实能力对齐了。
  • 报错信息也改进了：从写死「只有 thinker」改成动态列出「当前支持哪些阶段」，以后再加阶段不用改文案。

改动 B：新增两个命令行开关 —— cli/serve.py

这里做了两件事：(1) 在 serve() 函数签名里声明两个新 flag；(2) 在 apply_parallelism_cli_overrides() 里把这两个 flag 真正应用到配置上。

sglang_omni/cli/serve.py —— 新的 CLI 选项声明

概念 · Typer / Annotated 选项

它是什么：typer 是一个把 Python 函数自动变成命令行工具的库。函数的每个参数就是一个 CLI flag，Annotated[类型, typer.Option(...)] 用来描述这个 flag 的名字、别名、帮助文本。
为什么需要它：让「加一个命令行参数」=「给函数加一个带注解的形参」，省去手写参数解析。
在这个 PR 里：新增的两个 flag 默认值都是 None——这是关键设计：不传就等于完全不碰原配置（见下方测试 4），保证向后兼容。同时提供了两种写法（--image-encoder-tp-size 和下划线版 --image_encoder_tp_size），照顾不同用户习惯。

真正干活的逻辑在 apply_parallelism_cli_overrides() 里新增的这一段（它几乎是 thinker 处理逻辑的镜像复制）：

sglang_omni/cli/serve.py —— 应用 image_encoder 覆盖
# 1) 如果用户传了 --image-encoder-gpus，先把字符串 "4,5" / "[4,5]" 解析成 [4,5]
image_encoder_gpu_override = (
    _parse_gpu_placement("image_encoder_gpus", image_encoder_gpus)
    if image_encoder_gpus is not None else None
)
# 2) 只要 tp_size 或 gpus 任一被指定，才进入覆盖流程（否则原样不动）
if image_encoder_tp_size is not None or image_encoder_gpu_override is not None:
    image_encoder_stages = _find_matching_stages(           # 3) 在流水线里找到名为 image_encoder 的阶段
        pipeline_config, stage_name="image_encoder",
        reason="tensor parallel settings",
    )
    for stage in image_encoder_stages:
        if image_encoder_tp_size is not None:
            stage.tp_size = int(image_encoder_tp_size)  # 4) 写入 tp_size
            stage.parallelism.tp = stage.tp_size            # 5) 同步另一处并行配置
        if image_encoder_gpu_override is not None:
            stage.gpu = image_encoder_gpu_override          # 6) 指定这些 rank 落在哪几张卡
        _validate_stage_parallelism_config(                 # 7) 校验：卡数必须== tp_size、不能重复等
            "image_encoder", stage.tp_size, stage.gpu
        )
        if stage.tp_size == 1 and isinstance(stage.gpu, list):
            stage.gpu = int(stage.gpu[0])              # 8) TP=1 时把 [4] 收敛成标量 4

逐块解释（假设你完全没读过这套代码）：

  1. 解析 GPU 列表：_parse_gpu_placement 把人类友好的输入（"4,5" 或 "[4, 5]"）转成 Python 列表 [4, 5]。如果只有一个数字就返回标量 4。
  2. 守卫条件：两个 flag 都没传时，整段跳过——这就是「不传 = 不动」的兼容性保证。
  3. 定位阶段：一个 Omni 流水线由很多 stage 组成（image_encoder、audio_encoder、thinker、talker……）。_find_matching_stages 按名字把 image_encoder 那个找出来；找不到就抛 BadParameter。
  4. 写 tp_size：告诉引擎「把这个视觉模型切成 N 份」。
  5. 双写 parallelism.tp：配置里有两个字段都描述 TP（一个是阶段级 tp_size，一个是嵌套的 parallelism.tp），必须同步，否则后续校验会发现两处不一致。这是历史配置结构留下的「冗余字段」，作者老老实实地两边都写。
  6. GPU 落位：stage.gpu = [4, 5] 表示「这两个 TP rank 分别用 4 号、5 号卡」。
  7. 原子校验：_validate_stage_parallelism_config 检查：TP>1 时必须给出列表、列表长度要正好等于 tp_size、不能有重复卡号。所以 tp=2 gpus=4（只给一张卡）会被拒绝（对应测试 3）。
  8. 标量收敛：当 TP=1 时，[4] 这种单元素列表会被「拍扁」成裸整数 4，让下游配置保持一致风格（对应测试 2）。
🛠 可复用的工程思想：镜像式扩展 + 原子校验

注意这段几乎是 thinker 那段的「复制 + 改名」。作者没有去抽象出一个通用的 _apply_stage_tp(stage_name, ...) 辅助函数，而是直接平铺第二份。这是一个有意识的权衡：当只有 2 个调用点、且逻辑还在演化时，重复往往比过早抽象更安全、更易读（即 WET<DRY 的「三次法则」——出现第三次再抽象）。另外，所有校验都在真正改动配置之前完成，保证「要么全部成功，要么完全不变」，不会留下半残配置进入启动流程。

4. 涉及的知识点（重点 · 小白友好）

概念 1 · Tensor Parallelism（张量并行，TP）

它是什么：把一个神经网络层（比如一个巨大的矩阵乘法 / 注意力头）按维度切成 N 份，分别放到 N 张 GPU 上同时算，算完再用集合通信（如 all-reduce / all-gather）把结果拼起来。它切的是「一层内部的张量」，所以叫张量并行。
为什么需要它：① 当模型太大、单卡显存装不下时，TP 让多卡共同承载一份权重；② 即使装得下，多卡并行计算也能降低单次前向的延迟。
在这个 PR 里：视觉编码器（ViT）的注意力和线性层被切到 tp_size 张卡上。例如 --image-encoder-tp-size 2 --image-encoder-gpus 5,7 就是把 ViT 切两半，分别跑在 5 号和 7 号 GPU 上。

🍕 类比

TP 像「两个厨师合做同一张披萨」——一人擀左半边、一人擀右半边，最后合到一起。区别于流水线并行（Pipeline Parallelism）：那是「厨师 A 做饼底 → 传给厨师 B 铺料」，切的是层与层之间。SGLang Omni 的多阶段（encoder→thinker→talker）本质就是一种流水线，而 TP 是在某一个阶段内部再做切分——两种并行可以叠加（PR 描述里 thinker TP4 + image_encoder TP2 同时开就是例子）。

概念 2 · 多阶段流水线 / Stage（SGLang Omni 的核心架构）

它是什么：一个「全模态（Omni）」模型要处理图像、视频、音频、文本，并能生成文本和语音。SGLang Omni 把它拆成若干独立的 stage（阶段），每个 stage 是一个可单独配置、单独放卡、单独并行的组件，按数据流串成一条流水线：
image_encoder / audio_encoder（把图/音转成 embedding）→ thinker（语言主干，做理解与推理）→ talker（生成语音 token）→ code2wav（把语音 token 还原成波形）。
为什么需要它：不同 stage 的计算特性差异巨大——编码器是「一次性吃完输入」的非自回归计算，thinker/talker 是「一个 token 一个 token 吐」的自回归计算。拆开后可以各自独立放卡、独立调并行度、独立优化显存，资源利用率更高。
在这个 PR 里：正因为 image_encoder 是一个独立 stage、跑在独立进程，它才需要自己初始化一套 TP 上下文（见概念 3），也才需要一个专属的 CLI flag。这条 PR 本质是「让流水线里的某一个阶段获得独立的并行度控制权」。

概念 3 · 为什么图像编码器要「自己」初始化 TP 上下文

背景代码（来自 image_encoder.py 的 _init_sglang_tp，本 PR 没改它，但正是它让这次「放开」有意义）：

# 关键注释（原文）：多进程模式下，image encoder 和 thinker
# 跑在不同进程里，所以 image encoder 不能依赖 thinker 的
# tensor-parallel 状态，必须在请求时维持自己这一套 TP 上下文。
if not parallel_state.model_parallel_is_initialized():
    parallel_state.init_distributed_environment(
        backend="nccl", world_size=tp_size, rank=tp_rank, local_rank=0,
    )
    parallel_state.initialize_model_parallel(
        tensor_model_parallel_size=tp_size,    # 建立 TP 通信组
    )
    cls._did_init_tp = True
dp._ATTN_TP_SIZE = tp_size                     # 告诉注意力层：你现在被切成了 tp_size 份
dp._ATTN_TP_RANK = tp_rank

它是什么：SGLang 的并行层（如 VisionAttention、各种 ParallelLinear）在前向时会去读全局的「我是第几个 rank / 一共几个 rank」状态来决定切哪一片张量。这套状态由 torch.distributed + SGLang 的 parallel_state 维护，底层用 NCCL（NVIDIA 的多卡通信库）做 all-reduce 等集合通信。
为什么需要它：因为 image_encoder 是独立进程，它启动时进程里还没有任何 TP 状态（thinker 的 TP 状态在另一个进程，跨不过来）。所以它必须自己 init_distributed_environment + initialize_model_parallel，否则那些并行层不知道该怎么切。
在这个 PR 里：底层这套早就写好了（#553），但校验白名单（#513）把它锁死了。本 PR 做的就是「解锁」——让用户传进来的 tp_size/tp_rank 能真正流到这段初始化代码里。

概念 4 · 视觉编码器内部：ViT → Projector → L2 归一化

看一眼 image_encoder 真正算什么（_encode 方法）：

with torch.no_grad():
    embeds = self.visual(pixel_values, grid_thw)   # ① ViT：把图像 patch 编码成视觉特征
    if self.visual.use_deepstack:                    # （可选）多层特征堆叠，只取 base 部分
        embeds = embeds[:, : self.visual.image_emb_dim]
    embeds = self.linear_proj(embeds)              # ② Projector：把视觉维度映射到 LLM 的隐藏维度
    embeds = F.normalize(embeds, dim=-1)           # ③ L2 归一化：把向量长度统一为 1
  • ViT（Vision Transformer）：把图片切成一个个小块（patch），当成「视觉单词」序列喂进 Transformer，输出每个 patch 的特征向量。grid_thw 记录了 (时间帧 t, 高 h, 宽 w) 三个维度的 patch 网格大小（视频会有 t>1）。
  • Projector（投影层）：ViT 输出的维度（vision_dim）和语言模型的隐藏维度（llm_dim）通常不一样。Projector 是一个小 MLP，负责「翻译」——把视觉特征对齐到语言模型能直接吃的空间。这是多模态对接最关键的一环：让图像「说同一种语言」后注入 thinker。
  • L2 normalize：把每个 embedding 缩放到单位长度，稳定后续注入时的数值尺度。
  • token_counts 计算：(t*h*w) // (spatial_merge_size**2)——相邻 patch 会做空间合并（merge）以减少 token 数，这里算出每张图最终占多少个「视觉 token」，thinker 需要据此在序列里给图像留出对应数量的占位符。

和本 PR 的关系：TP 切的就是这里的 self.visual（ViT 内部的注意力和线性层）。图越大、ViT 越重，TP 带来的「单图编码提速 + 显存分摊」收益就越明显——这正是作者在 H100 上用 thinker TP4 + image_encoder TP2 跑通 cars.jpg 验证的场景。

5. 我能学到什么 / 延伸思考

  • 「运行时能力」和「对外接口」是两层皮。一个功能能跑（runtime）≠ 用户能用（CLI/校验放行）。成熟项目常常分阶段落地：先在底层实现并验证（#553），再单独开 PR 把它「暴露」出来（#687）。这种切分让每个 PR 都小而可审。
  • 校验白名单是一把双刃剑。它能防止用户踩进未实现的功能（保护性），但一旦底层能力更新而白名单忘了同步，就会出现「明明能跑却被拒」的矛盾。启示：能力清单（白名单）应该尽量和真实实现「单一数据源」，或至少有测试把两者绑在一起（本 PR 的 test_validate_stage_tp_support_* 就是这个作用）。
  • 向后兼容靠「默认 None + 守卫跳过」实现。新增参数默认 None、不传就完全不进入覆盖分支——这个模式值得记住，是给现有 CLI 加功能时最安全的做法。
  • 测试反转是行为变更的「自证」。把一条「断言会报错」的测试改成「断言成功」，diff 本身就完整记录了契约的变化，比写在 commit message 里更可靠。
  • 想深入，建议补的前置知识：
    1. 分布式基础：torch.distributed、NCCL、all-reduce / all-gather 这几个集合通信原语在做什么。
    2. 张量并行 vs 流水线并行 vs 数据并行的区别（Megatron-LM 论文是经典入门）。

▶ #714 · 修复 Omni CI 重跑时「实时标签门禁」失效的问题
作者 @Ratish1 · 状态 ✓ Merged（2026-06-09 17:35 UTC）· 查看 PR →

1. PR 概览

  • 标题：[CI] Fix live label gate for Omni CI reruns
  • 编号：#714　作者：@Ratish1（COLLABORATOR）　状态：已合并 merged
  • 链接：github.com/sgl-project/sglang-omni/pull/714

一句话概括：修复一个 GitHub Actions 的经典陷阱——「重跑（rerun）会复用旧的事件数据」。导致用 /tag-and-rerun-ci 给 PR 打上 run-ci 标签后，重跑却仍然看不到这个新标签，于是 Omni CI 被错误地跳过。修法是：在 CI 运行时实时去 GitHub API 查询 PR 的最新标签和草稿状态，而不是读事件里的旧快照。

2. 它解决了什么问题

这个 PR 不涉及模型架构，但它揭示的是一个所有用 GitHub Actions 的团队迟早会撞上的坑，很值得理解。

概念 · GitHub Actions 的「事件 payload 快照」陷阱

它是什么：当一个 workflow 被某个事件（如 pull_request）触发时，GitHub 会把当时 PR 的状态冻结成一份 JSON 快照（叫 event payload），workflow 里写的 github.event.pull_request.labels 读的就是这份触发那一刻的快照。
关键陷阱：当你点「Re-run（重跑）」时，GitHub 不会重新生成快照，而是复用最初那一份。所以如果你在首次触发之后才加了标签，重跑时读到的还是「没有标签」的旧快照。

具体的痛点链条是这样的：

  1. 开发者开了一个 PR，此时还没有 run-ci 标签 → Omni CI 的 if 条件判断「没标签」→ 跳过所有昂贵的自托管 GPU 任务（合理）。
  2. 维护者用 /tag-and-rerun-ci 命令：它加上 run-ci 标签，并触发重跑。
  3. 但重跑复用了第 1 步的旧 payload——里面 labels 还是空的！于是 contains(github.event.pull_request.labels.*.name, 'run-ci') 仍然为 false → CI 又一次被跳过。
  4. 结果非常诡异：PR 上明明挂着 run-ci 标签，CI 却死活不跑。维护者一脸问号。

3. 具体做了什么改动

核心思路：不再相信事件 payload 里的标签快照，而是在 job 运行时主动调用 GitHub API 去拿「此刻」的真实状态。所有改动都在 .github/workflows/ 下的 YAML 里。

改动 A：新增 opened 触发 + 把门禁挪进 preflight job

.github/workflows/omni-ci.yaml
on:
  pull_request:
    branches: [main]
-   types: [labeled, synchronize, reopened, ready_for_review]
+   types: [opened, labeled, synchronize, reopened, ready_for_review]

为什么加 opened：必须保证 PR 一开就有一次 workflow 运行存在，/tag-and-rerun-ci 才有东西可「重跑」。没有这次初始运行，就没有 run 可以 rerun。

然后把原来散落在每个 job 顶部的 if 标签检查，集中改写成 preflight job 里的几个步骤，用 actions/github-script 实时查询：

.github/workflows/omni-ci.yaml —— preflight 实时查询
- name: Fetch latest PR gate state
  if: ${{ github.event_name == 'pull_request' }}
  id: pr
  uses: actions/github-script@v7
  with:
    script: |
      // 重跑会复用原始 payload，所以这里主动调 API 拿“现在”的标签/草稿状态
      const pr = await github.rest.pulls.get({
        owner: context.repo.owner,
        repo: context.repo.repo,
        pull_number: context.issue.number,
      });
      core.setOutput("labels", JSON.stringify(pr.data.labels.map((l) => l.name)));
      core.setOutput("draft", pr.data.draft ? "true" : "false");

- name: Block draft PR
  if: ${{ github.event_name == 'pull_request' && fromJson(steps.pr.outputs.draft) }}
  run: echo "PR is draft. Blocking Omni CI." && exit 1

- name: Require run-ci label
  if: ${{ github.event_name == 'pull_request' }}
  run: |
    if [[ "${{ contains(fromJson(steps.pr.outputs.labels), 'run-ci') }}" == "false" ]]; then
      echo "Missing required label 'run-ci'. ..." && exit 1
    fi

逐块解释：

  • actions/github-script@v7 是 GitHub 官方动作，让你直接在 workflow 里写一段 JS 调用 GitHub REST API。这里用 pulls.get 拿到实时的 PR 对象。
  • core.setOutput("labels", ...) 把实时标签列表存成步骤输出，后续步骤用 steps.pr.outputs.labels 读取。
  • 「Block draft PR」：草稿 PR 直接 exit 1 让 preflight 失败——草稿不该烧 GPU。
  • 「Require run-ci label」：实时标签里若不含 run-ci 就失败。这次读的是真·此刻的标签，重跑也准。

改动 B：下游 job 改成「依赖 preflight 成功」而非各自重复检查

改动前（每个 job 各自判断旧 payload）改动后（统一信任 preflight 结果）
if: >-
  github.event_name == 'workflow_dispatch' ||
  (!github.event.pull_request.draft &&
   contains(github.event.pull_request
     .labels.*.name, 'run-ci'))
 
if: needs.preflight.result == 'success'

这一步是整个 PR 的「优雅」所在：原来每个昂贵的 GPU job（setup、tts-ci、qwen3-omni-ci、pr-test……）都重复粘贴同一段读旧 payload 的标签检查——既冗余又全都带着同一个 bug。改后它们只问一句：「preflight 这道门过了吗？」门禁逻辑收敛到唯一一处（single source of truth）。

同时，PR 还从 Qwen3 的可复用 workflow（test-qwen3-omni-ci.yaml）里删掉了 11 个 stage（stage-1 ~ stage-11）各自重复的那段「冻结标签」检查——因为这个可复用 workflow 只会被顶层 Omni CI 调用，门禁已经在上游 preflight 把过了，下游再查纯属冗余且有 bug。

4. 涉及的知识点（重点 · 小白友好）

概念 1 · CI / 自托管 Runner / 标签门禁

它是什么：CI（持续集成）是「每次代码改动自动跑测试」的机制。这个仓库的 CI 跑在自托管 GPU runner（自己的 GPU 机器）上，因为要真跑大模型推理，每次都很烧钱烧卡。
为什么需要标签门禁：正因为贵，团队不希望每个 PR 的每次提交都自动触发整套 GPU 测试。于是用一个 run-ci 标签当「开关」：只有维护者确认值得跑、打上标签，CI 才会真正启动。这是成本控制手段。
在这个 PR 里：问题就出在这个「开关」的读取方式上——读快照 vs 读实时。

概念 2 · job 依赖 / needs / 门禁收敛

它是什么：GitHub Actions 里 needs: preflight 表示「我要等 preflight 跑完才开始」，而 needs.preflight.result == 'success' 表示「只有 preflight 成功我才运行」。
为什么这样设计更好：把「该不该跑」的判断只写在一个地方（preflight），其他 job 只是顺着依赖链信任它。这避免了「同一段逻辑复制十几遍、改的时候漏改某一处」的维护灾难。
在这个 PR 里：11+ 个 stage 的重复 if 被一句 needs.xxx.result == 'success' 取代。

🎫 类比

旧设计像「演唱会每个入口的保安都各自拿着一份开场前打印的嘉宾名单查票」——你中途补办了 VIP（加标签），但保安手里的名单是旧的，照样拦你。新设计是「门口设一个总验票口（preflight），现场连后台系统实时核验（调 API），通过了发一个手环；里面所有区域的保安只认手环（needs.result == success），不再各自查名单」。又快又不会出错。

5. 我能学到什么 / 延伸思考

  • 分布式系统的「读快照 vs 读实时」是个永恒主题。事件驱动系统里，事件携带的数据是「那一刻的快照」，一旦你的逻辑依赖「最新状态」，就必须主动去权威数据源重新拉取。这个教训在消息队列、Webhook、缓存失效等场景反复出现。
  • 消除重复 = 消除 bug 的藏身之处。同一段门禁逻辑被复制十几份时，bug 也被复制了十几份。把它收敛到单一 preflight，不仅好维护，更是从结构上让 bug 无处可藏。
  • CI 成本治理是真实工程问题。GPU CI 很贵，「标签 + 重跑」这种人工闸门是常见的省钱模式；理解它能帮你设计自己项目的 CI 触发策略。
  • 想深入，建议补的前置知识：GitHub Actions 的事件类型与 payload 结构、actions/github-script 与 REST API、workflow 的 needs / if / 表达式语法（fromJson、contains）、可复用 workflow（reusable workflow）的调用机制。
