先建立地图：SGLang 与 SGLang-Omni 到底是什么？

看不懂下面的 PR？先花 3 分钟读这一节，把术语地基打好。

SGLang 是一个开源的大模型推理（inference）引擎——也就是"把训练好的模型跑起来、对外提供服务"的那层基础设施。它的卖点是高吞吐：靠 RadixAttention（前缀缓存复用）、continuous batching（连续批处理）、CUDA Graph（算子图捕获）等手段把 GPU 榨干。

SGLang-Omni 是在 SGLang 之上做的多模态 / 全模态（Omni）扩展。"Omni"意味着它要同时处理文本、图像、音频，甚至直接输出语音（TTS）。这类模型不是"一个网络一口气算完"，而是被拆成多个阶段（stage）串成一条流水线（pipeline）。

🧩 关键心智模型：多阶段流水线（multi-stage pipeline）

以 Qwen3-Omni 语音管线为例，一句"看图说话再朗读"的请求会依次流过这些阶段：

# 8-stage speech pipeline（简化示意）
preprocessing  # 预处理：把图片/音频切成模型能吃的张量
  → image_encoder / audio_encoder   # 多模态编码器：把像素/声波编码成 embedding
  → thinker            # "思考者"：核心大语言模型，理解输入并产生文本/语义
  → talker_ar          # "说话者"：自回归地把语义转成离散语音 token
  → code2wav           # 声码器：把语音 token 还原成波形 wav

每个阶段都是独立的进程 / GPU 角色，阶段之间用消息（payload）传递数据。今天的两个核心 PR 分别动了其中两个关键位置：thinker（#771）和各类生成阶段（#843）。

📖 名词速查（后文反复出现）
  • thinker / talker：Omni 架构里的两个角色。thinker = 负责"理解与思考"的主干 LLM；talker = 负责"把思考结果说出来"的语音生成模型。源自 Qwen-Omni 系列的设计命名。
  • stage（阶段）：流水线里的一个独立计算单元，有自己的进程、GPU、配置（StageConfig）。
  • TP / tensor parallelism（张量并行）：把一个大模型的权重横切到多张 GPU 上同时算，再把结果拼回来。详见 #771。
  • CUDA Graph：把一连串 GPU 操作"录制"成一张可重放的图，省去每次重新下发指令的开销。详见 #843。
  • ServerArgs：SGLang 启动一个推理引擎时的"参数表"对象，几乎所有调优旋钮都挂在它上面。
#771

[Qwen3-Omni] TP correctness — 让多进程 TP 启动的 thinker 自动关闭"自定义 all-reduce"

作者 luojiaxuan ｜ 状态 merged（2026-06-21）｜ 查看 PR #771 ↗

1 · PR 概览

一句话：把"当 thinker 阶段用 >1 张 GPU 做张量并行时，要关掉 SGLang 的自定义 all-reduce 内核"这条规则，从每个模型各写一遍、且只在示例脚本里生效，下沉成所有多模态配置共享的一个声明式开关，从而让正式的 sglang_omni serve 启动路径也能自动正确生效。这是一个正确性（correctness）修复，回归守卫的是 issue #760。

2 · 它解决了什么问题

背景要从"张量并行"和"all-reduce"说起（下面"知识点"会展开）。简单讲：当 thinker 被切到多张 GPU 上时，每算完一层都需要把各卡的部分结果加总同步一次，这个操作叫 all-reduce。SGLang 为了快，默认用一个自己手写的 CUDA all-reduce 内核（custom all-reduce）替代通用的 NCCL。

问题是：这个自定义内核对硬件互联拓扑（NVLink / PCIe P2P）有要求，在某些多进程 TP 启动方式下会算出错误结果或直接挂掉。团队早就知道要对 thinker 加一句 disable_custom_all_reduce=True 来回退到稳妥实现——但这段逻辑散落在各个模型里、而且实测只在"示例脚本"路径生效，真正的 serve 命令行启动路径漏掉了 Qwen3-Omni（issue #760 的现象）。结果就是：用户用官方 demo 跑没问题，一旦用正式服务命令开 TP>1，thinker 就可能出错。

类比：就像一栋楼里每层楼都各自贴了一张"地震时走安全通道"的告示（重复、容易贴漏），而且正门的告示忘了贴。这个 PR 做的是：把告示改成楼栋统一的电子广播——一处声明，全楼（含正门）自动生效。

3 · 具体做了什么改动

3.1 核心文件
  • sglang_omni/config/schema.py — 流水线配置基类 PipelineConfig，新增共享开关。
  • sglang_omni/models/qwen3_omni/config.py — Qwen3-Omni 各管线配置。
  • sglang_omni/models/ming_omni/config.py — Ming-Omni 各管线配置（把已有逻辑改成复用新机制）。
  • tests/unit_test/qwen3_omni/test_pipeline.py — 新增 3 个回归测试。
3.2 改动前后逻辑对比

改动前：每个模型配置类各自重写（override）一个方法 tensor_parallel_server_args_overrides，里面硬编码同一段 if 判断：

# 改动前：Ming / Qwen3-Omni 的每个管线类里都重复这一段
class MingOmniPipelineConfig(PipelineConfig):
    @classmethod
    def tensor_parallel_server_args_overrides(cls, *, stage_name, tp_size):
        if stage_name == THINKER_STAGE and tp_size > 1:
            return {"disable_custom_all_reduce": True}
        return {}

改动后：在基类 PipelineConfig 上加一个声明式的类变量，并把判断逻辑只写一次放进基类方法：

# schema.py —— 基类新增一个"哪些阶段要关自定义 all-reduce"的声明位
class PipelineConfig(BaseModel):
    tensor_parallel_disable_custom_all_reduce_stages: ClassVar[tuple[str, ...]] = ()
                                                       # ↑ 默认空：什么都不关

    @classmethod
    def tensor_parallel_server_args_overrides(cls, *, stage_name, tp_size):
        if tp_size > 1 and stage_name in cls.tensor_parallel_disable_custom_all_reduce_stages:
            return {"disable_custom_all_reduce": True}   # ← 命中：注入这条 ServerArgs 覆盖
        return {}

然后各模型只需一行声明就接入了这套机制（再也不用重写方法）：

# qwen3_omni/config.py —— 新增一个共享基类，所有 Qwen3-Omni 管线都继承它
class _Qwen3OmniBasePipelineConfig(PipelineConfig):
    architecture: ClassVar[str] = "Qwen3OmniMoeForConditionalGeneration"
    tensor_parallel_disable_custom_all_reduce_stages: ClassVar[tuple[str, ...]] = (
        THINKER_STAGE,     # ← 唯一需要写的"业务知识"：thinker 阶段要关
    )

class Qwen3OmniPipelineConfig(_Qwen3OmniBasePipelineConfig): ...        # 6 阶段纯文本
class Qwen3OmniSpeechPipelineConfig(_Qwen3OmniBasePipelineConfig): ...  # 8 阶段语音

Ming-Omni 这边同理：抽出 _MingOmniBasePipelineConfig，把原本在 3 个管线类（文本 / 语音 / 流式语音）里重复 3 遍的 override 删掉，统一继承基类。净效果是删代码 + 修 bug：Qwen3-Omni 之前根本没有这段保护（所以 #760 才会发生），现在它和 Ming 走同一条受测试守卫的路径。

3.3 测试怎么验证它真的修好了

关键的回归测试断言了两件事：(a) 6 个管线类都不再自己重写那个方法（确保走的是共享 hook），(b) thinker 在 tp=2 时确实拿到 disable_custom_all_reduce=True，而 tp=1 或别的阶段（encoder/talker/code2wav）拿到空字典：

for cls in classes:
    assert "tensor_parallel_server_args_overrides" not in cls.__dict__   # 没有自己 override
    assert cls.tensor_parallel_server_args_overrides(
        stage_name="thinker", tp_size=2) == {"disable_custom_all_reduce": True}

还有一个端到端测试，模拟真实的 CLI serve 路径（apply_parallelism_cli_overrides → _apply_tensor_parallel_server_args_overrides），断言写进 thinker 阶段的 server args 里真的带上了这个开关——这正是 #760 漏掉的那条路径。

4 · 涉及的知识点（重点展开）

知识点 A · 张量并行 Tensor Parallelism (TP)

它是什么 → 一种把单个模型拆到多张 GPU 的并行方式。具体是把权重矩阵按行或列切开，每张卡只存一部分、只算一部分。比如一个 [4096×4096] 的线性层，TP=2 时每张卡各拿 [4096×2048]。

为什么需要它 → 大模型一张卡放不下，或放得下但算得慢。TP 让多卡像"一张大卡"一样协同，显著降低单请求延迟。它和"数据并行（多卡各跑不同请求）"是正交的两个维度。

在这个 PR 里 → 只有当 thinker 的 tp_size > 1（真的切到多卡了）才需要这层修复；tp_size == 1 时根本没有跨卡同步，自然不触发。

类比：TP 像把一道大菜分给 2 个厨师，一人切菜一人炒——但每过一步两人要把半成品对齐合并一次，这个"合并"就是下面的 all-reduce。
知识点 B · All-Reduce 与"自定义 all-reduce 内核"

它是什么 → All-reduce 是分布式计算里的集合通信原语：把 N 张卡上的张量逐元素求和（或求平均），再把同一个结果广播回每张卡。TP 里每个被切开的层算完都要 all-reduce 一次来"拼回完整结果"。

为什么需要它 → 因为每张卡只算了部分和（partial sum），必须加总才能得到正确的整层输出。标准实现是 NVIDIA 的 NCCL 库；SGLang 额外提供一个手写的 custom all-reduce CUDA 内核，在小数据量、有 NVLink/P2P 直连时延迟更低。

在这个 PR 里 → 这个自定义内核依赖特定的 GPU 间互联拓扑（P2P / NVLink），在多进程 TP 启动时不一定满足，会导致结果错误或崩溃。所以对 thinker 注入 disable_custom_all_reduce=True，回退到稳妥但稍慢的通用 all-reduce——用一点点性能换正确性。PR 描述里特意保留了"拓扑门控（topology gate）"的铺垫：未来若能确认是 NVLink mesh，再选择性放开。

知识点 C · 设计模式：声明式配置 + 模板方法 (Template Method)

它是什么 → 把"会变化的业务知识"（哪些阶段要关）抽成一个数据声明（类变量 ClassVar 元组），把"不变的处理流程"（怎么判断、返回什么）放在基类方法里。子类只填数据、不重写逻辑。

为什么需要它 → 改动前 6 处复制粘贴同一段 if，任何一处漏改/写错就是一个潜在 bug（#760 正是漏了一处）。"逻辑只写一次、数据各自声明"消除了这种重复，也让测试可以统一守卫。

在这个 PR 里 → ClassVar[tuple[str, ...]] 是声明位，基类的 tensor_parallel_server_args_overrides 是模板方法。这是"用数据驱动行为、把策略上提到基类"的典型重构。

补充名词：ServerArgs overrides 指的是"启动 SGLang 引擎前，往参数表里塞的覆盖项"——TP 设置会隐含一些必须的引擎参数，这个方法就是把这种隐含规则显式化。

5 · 我能学到什么 / 延伸思考

  • 正确性 bug 往往藏在"重复代码 + 多条入口路径"的缝里：示例脚本对，serve 路径错——这类"看起来都实现了却漏一条路径"的 bug，根因常常是逻辑被复制了多份。消重 + 端到端测试覆盖每条入口是治本之道。
  • 性能优化要可降级：custom all-reduce 是优化项，但工程上必须能一键回退到稳妥实现。"快"永远不能以"偶尔算错"为代价。
  • 最小化 PR 的纪律：作者明确把这个 PR "瘦身"成只含正确性修复，把 prefill/coalescing/idle-broadcast 等调优机器全部剥离到后续 PR。一个 PR 只做一件事，便于审查与回滚。
  • 延伸前置知识：集合通信（all-reduce/all-gather/reduce-scatter）、NCCL 与 NVLink 拓扑、Megatron 式 TP 的行/列切分、以及"流水线并行 PP vs 张量并行 TP vs 数据并行 DP"三者的区别。
#843

[TTS] Validate generation batch policies — 给所有音频生成阶段建立统一的"批处理策略校验器"

作者 luojiaxuan ｜ 状态 merged（2026-06-22）｜ 查看 PR #843 ↗ ｜ 关联 issue #836（Workstream 1）

1 · PR 概览

一句话：新增一个可复用的 generation_batch_policy 模块，统一构造和校验所有 SGLang 后端生成阶段（7 个 TTS / 音频模型）的批处理参数——CUDA Graph 的捕获桶、max_running_requests、torch.compile 上限、以及模型侧实际分配的缓冲区容量，必须互相覆盖、对得上，否则启动时直接报错。

2 · 它解决了什么问题

这些音频生成阶段启动时要设一堆相互关联的批处理旋钮，之前的痛点有三：

  • 魔法数字散落、容易不一致：每个模型各自硬编码一份 cuda_graph_bs = [1, 2, 4, 8, 16] 这样的桶列表，和 cuda_graph_max_bs、max_running_requests 之间没人保证对得上。
  • "静默错配"很危险：如果允许的并发请求数（max_running_requests=64）超过了 CUDA Graph 捕获的最大批量（cuda_graph_max_bs=16），或者超过了模型实际分配的 KV/采样缓冲区行数，运行时要么走慢速回退、要么越界——而且不会在启动时报错，等到线上高并发才暴雷。
  • 旧的"部分校验"旋钮形同虚设：原来有一些 partial/require 校验开关，但默认不强制，等于没护栏。

这个 PR 把"批处理参数必须自洽"变成启动期的硬约束（fail-fast），同时把 S2-Pro 的 torch_compile_max_bs 从 16 提到 64，使三个上限（max_running_requests / cuda_graph_max_bs / torch_compile_max_bs）齐平到 64，扩大了高并发下的图覆盖。

类比：之前像一栋楼的"电梯限载 16 人"但"门禁却放 64 人进来排队"——平时没事，人一多就卡死或出危险。这个 PR 是在开业前装了一个验收员：限载、门禁、楼道宽度三者对不上，就不准开业。

3 · 具体做了什么改动

3.1 核心新增：scheduling/generation_batch_policy.py（+196 行）

这个新模块提供三个工具函数 + 一份报告数据类：

(a) build_default_cuda_graph_bs(max_bs) — 生成"标准捕获桶列表"

def build_default_cuda_graph_bs(max_bs: int) -> list[int]:
    values = [1, 2, 4, 8, 12]
    values.extend(range(16, 257, 8))    # 16,24,32,...,256：低区间密一点
    values.extend(range(272, 512, 16))   # 272,288,...：中区间疏一点
    values.extend(range(512, max_bs + 1, 32))  # 512 以上更疏
    values = [bs for bs in values if bs <= max_bs]
    if not values or values[-1] != max_bs:
        values.append(max_bs)                  # 保证最大值一定被捕获
    return values

它生成的桶刻意对齐 SGLang 默认的捕获桶（小批量密集、大批量稀疏），这样校验不会人为缩小图覆盖。例如 max_bs=64 → [1,2,4,8,12,16,24,32,40,48,56,64]。

(b) sync_cuda_graph_bs_with_max_bs(...) — 用户改了 max 就自动重算桶

def sync_cuda_graph_bs_with_max_bs(overrides, server_args_overrides):
    if "cuda_graph_max_bs" in server_args_overrides \
       and "cuda_graph_bs" not in server_args_overrides:
        # 用户只改了上限、没给具体桶 → 按新上限重建一份桶
        overrides["cuda_graph_bs"] = build_default_cuda_graph_bs(
            int(overrides["cuda_graph_max_bs"]))

解决一个很隐蔽的坑：用户/上层只覆盖了 cuda_graph_max_bs，却忘了同步 cuda_graph_bs 列表，导致"上限说 64、桶里最大才 16"的不一致。

(c) validate_generation_batch_policy(...) — 启动期总校验

它把所有不变量集中检查，任一不满足就抛 ValueError。核心约束（逐条翻译）：

# CUDA Graph 开着时：
- cuda_graph_max_bs / cuda_graph_bs 必须显式给出（不能靠隐式默认）
- max(cuda_graph_bs) 必须 == cuda_graph_max_bs        # 桶顶要等于声明的上限
- cuda_graph_max_bs >= max_running_requests           # 图要能覆盖最大并发
# 只有 enable_torch_compile=True 时才强制：
- torch_compile_max_bs >= max_running_requests        # 编译上限也要覆盖并发
# 若传入模型实际缓冲容量：
- model_buffer_bs >= max_running_requests             # 物理 buffer 要装得下并发

还有一个 _normalize_cuda_graph_bs 辅助函数，校验桶列表是严格递增、全为正整数、非空——把"手滑写错列表"也挡在门外。

3.2 在 7 个生成阶段统一接入

改动模式高度一致，每个 stages.py 都是"构造时用默认桶 + 启动后调校验器"。以 MOSS-TTS 为例，原本的魔法列表被替换：

- "cuda_graph_bs": [1, 2, 4, 8, 16],
+ "cuda_graph_bs": build_default_cuda_graph_bs(16),
  ...
  if server_args_overrides:
      overrides.update(server_args_overrides)
+     sync_cuda_graph_bs_with_max_bs(overrides, server_args_overrides)
  ...
+ validate_generation_batch_policy(model_name="MOSS-TTS", server_args=server_args)

覆盖的阶段：Higgs TTS、MOSS-TTS、MOSS-TTS Local、Qwen3-TTS、Voxtral TTS、FishAudio S2-Pro、Qwen3-Omni talker_ar。其中两个还把"模型物理缓冲容量"也喂给校验器做更强的检查：

  • Higgs TTS 暴露 sampler_pool_max_running_requests（采样器池的行数）作为 model_buffer_bs。
  • FishAudio S2-Pro 取 min(VQ 解码 buffer 行数, 音频解码器 KV cache 的 batch 维) —— 即同时受两个张量第 0 维容量约束，取较小者：
def _resolve_s2pro_model_buffer_bs(model) -> int:
    return min(
        int(model.vq_decode_max_batch_size),               # VQ 码本解码缓冲
        int(model._audio_decoder.kv_cache_max_batch_size), # = kv_cache.k_cache.shape[0]
    )

S2-Pro 同时把 cuda_graph_bs/max_bs 设到 64、torch_compile_max_bs 16→64，让 Fast-AR 编译批量和图覆盖都齐到 64（PR 实测启动日志确认捕获桶为 [1,2,4,8,12,16,24,32,40,48,56,64]）。

3.3 性能：校验不是免费午餐，但几乎零开销

4 · 涉及的知识点（重点展开）

知识点 A · CUDA Graph 与"捕获桶 (capture buckets)"

它是什么 → 正常跑 GPU 时，CPU 要一条一条地把 kernel"下发（launch）"给 GPU，每条下发都有微秒级开销。CUDA Graph 把"一次前向里的整串 kernel 调用"录制成一张静态图，之后直接"重放（replay）"整张图，省掉逐条下发的 CPU 开销。

为什么需要它 → 自回归解码每步只算 1 个 token、计算量很小，瓶颈往往不是算力而是 kernel launch 开销（launch-bound）。CUDA Graph 能让小批量解码提速可观（MOSS-TTS Local 注释里就提到 eager 模式 ~22ms/帧、纯属 launch-bound）。

"桶"是怎么回事 → CUDA Graph 是形状静态的：录制时 batch size 多大，重放时就得多大。但线上并发是变化的（这一刻 3 个请求、下一刻 30 个）。所以要预先为若干个固定批量各录一张图——这些预设批量就是"桶"。运行时把当前并发向上取整到最近的桶用对应的图。桶太稀 → 浪费算力（小批量也用大图）；桶太密 → 录制慢、占显存。本 PR 的 [1,2,4,8,12,16,24,...] 就是在"覆盖度"和"成本"间取的平衡。

类比：CUDA Graph 像把一套固定动作预先排练并录像，演出时直接放录像而不是临场喊口令。但录像是"几人版"固定的，所以要为 1 人 / 2 人 / 4 人……各录一版（桶），来几个人就放最接近的那一版。
知识点 B · max_running_requests 与 Continuous Batching（连续批处理）

它是什么 → max_running_requests 是引擎同时在跑的请求数上限。连续批处理（又叫 in-flight batching）指：不像传统"凑满一批再一起算"，而是每个解码步都动态地把新到的请求塞进当前批、把已完成的请求踢出去，让 GPU 时刻保持高占用。

为什么需要它 → 不同请求生成长度差异巨大，静态批会被最慢的拖死。连续批处理把吞吐拉满，是现代推理引擎（vLLM/SGLang）的核心吞吐来源。

在这个 PR 里 → 校验器要求 cuda_graph_max_bs >= max_running_requests：既然最多可能 64 个请求并发，CUDA Graph 就必须能覆盖到 64，否则高并发那一刻没有对应的图可用，只能退化到慢速 eager。这就是"限载必须 ≥ 门禁放行量"的工程化表达。

知识点 C · KV Cache 与"模型缓冲区容量 (model_buffer_bs)"

它是什么 → Transformer 自回归解码时，前面 token 算出的 Key/Value 向量会被缓存复用（不必每步重算整段历史），这就是 KV Cache。它在显存里是预先分配好的固定大张量，第 0 维通常就是"能同时容纳多少个请求/序列"。

为什么需要它 → KV Cache 把解码从 O(n²) 降到每步 O(n)，是 LLM 推理能跑快的根基。但它占显存且容量固定，分配多大就只能装多少并发。

在这个 PR 里 → 校验器新增 model_buffer_bs >= max_running_requests 这条：即便参数表里写 max_running_requests=64，如果模型实际分配的 KV cache / 采样池 / VQ 解码缓冲只有 32 行，那也是骗人的——第 33 个请求会越界。S2-Pro 取两个缓冲张量 shape[0] 的较小值正是在问"物理上到底装得下几个"。这是把"配置声明"和"物理现实"对账的一步。

知识点 D · torch.compile 与 torch_compile_max_bs

它是什么 → torch.compile 是 PyTorch 2.x 的即时编译器，把 Python 算子图编译成融合后的高效 kernel。和 CUDA Graph 一样，它对输入形状敏感，需要为不同批量上限编译。

为什么只在开启时才校验 → PR 特意做到"torch_compile_max_bs 的覆盖只在 enable_torch_compile=True 时强制"。因为编译没开时这个参数无意义，强制它会误伤。这体现了"约束要跟随实际生效的特性"的细致。

在这个 PR 里 → S2-Pro 把它从 16 提到 64，使编译覆盖追上 64 的并发上限。

知识点 E · 工程思想：Fail-Fast 校验 + 单一事实来源 (SSOT)

它是什么 → Fail-Fast 指"错误配置在启动时立刻、清晰地报错，而不是潜伏到运行期才以诡异症状爆发"。SSOT 指"桶列表怎么生成只有 build_default_cuda_graph_bs 这一个出处"。

为什么需要它 → 推理基础设施的参数耦合极强（图/并发/缓冲/编译四者互相牵制），靠人工记忆对齐必然出错。把约束写成代码、启动即校验，把"线上半夜暴雷"提前成"启动 5 秒报错"。

在这个 PR 里 → 一个 196 行的小模块 + 7 处统一调用，就把过去散落的隐性契约固化成显式护栏，且自带 134 行单元测试覆盖每条约束分支。

5 · 我能学到什么 / 延伸思考

  • "配置正确性"也是产品质量：很多线上事故不是算法错，而是参数互不自洽。把"参数不变量"写成可测试的校验器，是高可靠系统的标配。
  • 声明 vs 现实要对账：max_running_requests 是"我想要"，KV cache/buffer 的 shape[0] 是"我实际有"。优秀工程会主动把两者对齐校验，而不是假设它们一致。
  • 性能改动要有可复现的基准：同机同容器、固定 commit、重复多次取中位数、剔除离群、不夸大收益——这套 benchmark 纪律本身就是可复用的方法论。
